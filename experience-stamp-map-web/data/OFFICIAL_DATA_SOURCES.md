# Official Data Source Plan

마이리얼트립 MCP 상품 데이터와 도시/국가 공식 데이터를 함께 쓰기 위한 데이터 소스 계획이다.

API별 실제 호출 방식은 `DATA_FETCHING_GUIDE.md`에 정리한다.

MVP에서 실제로 수집·노출할 데이터 범위는 `MVP_DATA_SCOPE.md`를 따른다. 이 문서의 소스 목록은 확장 후보까지 포함하지만, 1차 데모에서는 오사카·후쿠오카·부산 중심으로 축소한다.

## 1. 데이터 결합 원칙

### MyRealTrip MCP

역할:

- 예약 가능한 T&A 상품 확인
- 상품명, 카테고리, 가격대, 공식 상품 URL 확보
- 입장권/투어/패스/미식/액티비티 등 구매 방식 분리
- 실제 상품으로 연결 가능한 경험과 연결 불가능한 경험 구분

사용 기준:

- 공식 상품 URL이 있는 경험만 `예약 가능 경험`으로 표시한다.
- 상품 상세에서 명시된 포함사항과 일정만 긍정 근거로 쓴다.
- 불포함사항, 주변 언급, 제목의 과장 문구는 긍정 근거로 쓰지 않는다.

### Official city/country data

역할:

- 지역 공식 명소, 관광자원, 축제, 음식, 인증 식당, 시장, 야시장, 교통/지도 기준점 확보
- 마이리얼트립 상품은 적지만 도시 매력을 설명하는 경험 보강
- 상품화 후보, 파트너 발굴 후보, 콘텐츠 후보 탐색

사용 기준:

- 공식 데이터는 `지역 공식 자산` 또는 `상품화 후보`로 표시한다.
- 공식 데이터만으로는 예약 가능하다고 표현하지 않는다.
- 출처명, URL, 수집일, 라이선스, 갱신 주기를 저장한다.

## 2. MVP 우선 데이터 소스

### 한국 공통

1. 한국관광공사 TourAPI
   - 관광지, 행사/축제, 음식점, 시장, 공연 등 관광 콘텐츠 보강
   - 도시별 `뭐 하지`, `뭐 먹지` 기본 후보 생성에 사용

2. 공공데이터포털 관광식당 인허가/관광식당 데이터
   - 전국 지자체 관광식당 인허가 정보
   - 인증 식당/관광식당 후보 보강에 사용

### 서울

1. 서울 열린데이터광장
   - 서울시 오픈API 인증키 기반
   - 관광/문화/상권/실시간 도시데이터 보강 후보

2. 스마트서울맵 OpenAPI
   - 다국어 POI, 테마 지도, 주소/좌표 변환 후보
   - 서울 지도형 경험 보강에 사용 가능

### 부산

1. 부산광역시_부산맛집정보 서비스
   - 부산관광포털 비짓부산 미식투어 기반
   - 식당/카페명, 주소, 홈페이지, 전화번호, 좌표, 상세정보, 주변 관광지, 사진, 리뷰 제공
   - 한국어/영어/일본어/중국어 간체/번체 다국어 제공
   - 부산 `뭐 먹지`와 인증성 미식 후보에 최우선 사용

2. Big-데이터웨이브
   - 부산시 통합 공공데이터 창구
   - 관광/교통/지역 데이터 탐색 보조

### 대구

1. 대구광역시_맛집
   - 대구푸드 기반 맛집 API
   - 주소, 음식, 카테고리, 음식점명, 연락처, 영업시간, 좌석수, 주차장, 웹사이트, 가능언어, 예약가능여부, 메뉴 등 제공
   - 대구10미, 대구명품빵, 대구우수식품, 먹거리골목 데이터 포함
   - 대구 `뭐 먹지`, 인증 식당, 지역 대표 음식 브랜드 보강에 최우선 사용

2. 대구 관광데이터랩
   - 관광 빅데이터/실태조사/관광산업 분석 자료
   - 내부 사업개발 리포트의 시장성 근거로 사용

### 전주/여수

우선순위:

1. 한국관광공사 TourAPI로 관광지/음식점/행사 기본 후보 확보
2. 공공데이터포털에서 전주시/여수시 개별 관광·음식점 데이터 탐색
3. 지자체 공식 관광 포털의 추천 음식, 인증 음식점, 대표 관광지 목록을 수집 후보로 관리

주의:

- 전주/여수는 API 형태가 약할 수 있으므로, 최초 MVP에서는 공식 URL 기반 수동 seed도 허용한다.
- 수동 seed는 반드시 출처 URL과 수집일을 저장한다.

### 일본 공통

1. Google Maps Platform
   - 지도, 도보/대중교통 경로, 장소 좌표 보강
   - 실시간 위치추적이 아니라 사용자가 선택한 앵커 기반 경로 계산에만 사용

2. 도시/현 단위 오픈데이터
   - 일본은 관광 공식 데이터가 도시별로 분산되어 있으므로 도시별 카탈로그를 우선 확인한다.

### 도쿄

1. Tokyo Tourism Data Catalog
   - 도쿄도 관광 통계, 방문지, 소비, 관광시설 방문자 조사 로데이터
   - 내부 리포트와 대표/숨은 선택지 판단의 시장성 근거로 사용

2. Tokyo Metropolitan Open Data API Catalog
   - 관광객 수 등 실태조사 CSV/API
   - 관광지 방문 규모, 지역별 수요 분석에 사용

### 오사카

1. Osaka City Map Navi Open Data
   - 오사카시 지도/시설 정보 포인트 데이터
   - 공원, 스포츠 시설, 관공서 등 지도 기준점 확보

2. Osaka tourism official pages
   - 관광 명소/음식/패스 관련 공식 페이지를 seed 후보로 관리
   - API가 부족한 경우 출처 URL 기반 공식 자산 테이블로 보강

### 후쿠오카

1. Fukuoka City API Catalog
   - 후쿠오카시 API 카탈로그
   - 특히 `후쿠오카시 야타이 기본정보/개점정보`가 MVP에 유용
   - 야타이 이름, 주소, 위치, 메뉴, 영업 상태 등을 `뭐 먹지`와 야간 일정 앵커로 활용 가능

2. Fukuoka Prefecture Open Data
   - 현 단위 오픈데이터 사이트
   - 관광/문화/교통 데이터 탐색 보조

### 홋카이도

우선순위:

1. 일본 관광 공식 자료와 홋카이도/삿포로 오픈데이터 카탈로그 탐색
2. Google Maps 기반 좌표/동선 보강
3. 마이리얼트립 MCP 상품 데이터에서 투어 일정·포함 장소를 우선 추출

주의:

- 홋카이도는 지역 범위가 넓어 도시 단위(삿포로, 오타루, 비에이, 후라노 등)로 세분화해야 한다.

## 4. 매칭 로직

1. MCP 상품에서 경험 후보 추출
2. 공식 데이터에서 명소/음식/인증 식당 후보 추출
3. 이름, 별칭, 주소, 좌표, 카테고리로 매칭
4. 같은 경험이면 하나로 병합
5. MCP 상품 URL이 있으면 `예약 가능 경험`
6. 공식 데이터만 있으면 `지역 공식 자산`
7. 공식 데이터는 강하지만 MCP 상품이 적으면 `상품화 후보`

## 5. 적용 우선순위

### 1차 데모

1. MyRealTrip MCP 오사카/후쿠오카/부산 상품 데이터
2. 후쿠오카 야타이 BODIK/CKAN 공개 데이터셋
3. 오사카 Map Navi 문화·관광/명소/철도·버스 CSV 일부
4. 부산 공식 맛집/관광 데이터는 seed 수준으로 소량 사용

### 2차 확장

1. 부산 맛집 API
2. 대구 맛집 API
3. 한국관광공사 TourAPI
4. 공공데이터포털 관광식당 API
5. 도쿄 관광 데이터 카탈로그
6. 전주/여수 공식 관광 포털 seed
