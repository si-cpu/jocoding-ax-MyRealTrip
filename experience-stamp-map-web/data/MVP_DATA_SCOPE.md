# MVP Data Scope

이 프로젝트는 시장 출시용 데이터 플랫폼이 아니라 포트폴리오 MVP다. 따라서 “많이 수집하기”보다 “작은 데이터로 제품 구조를 선명하게 보여주기”를 우선한다.

## 1. 1차 데모 범위

### 데모 도시

1차 데모에서는 3개 도시만 깊게 보여준다.

- 오사카
- 후쿠오카
- 부산

나머지 MVP 도시(서울, 전주, 여수, 대구, 도쿄, 홋카이도)는 확장 가능 도시로 표시하되, 1차 데모에서 모든 데이터를 채우지 않는다.

## 2. 도시별 데이터 상한

각 도시의 사용자 화면에는 다음 상한을 둔다.

| 데이터 | 도시별 상한 | 이유 |
|---|---:|---|
| 대표 경험 | 3개 | 사용자가 도시의 핵심 매력을 빠르게 이해 |
| 숨은 선택지 | 3개 | “나다운 여행” 선택 폭 제공 |
| 먹거리 경험 | 3개 | `뭐 먹지`를 과하지 않게 보강 |
| 이동/숙소/역 앵커 | 3개 | 타임라인과 지도 구성에 필요한 정도만 사용 |
| 공식 자산 | 10개 | 공식 데이터가 너무 많아지는 것 방지 |
| 마이리얼트립 연결 상품 | 경험당 최대 5개 | 상품 피로도 방지 |

즉 1차 데모의 도시별 경험 수는 8~12개 정도로 제한한다.

## 3. 데이터 소스 축소 기준

### 오사카

사용:

- MyRealTrip MCP 상품 데이터
- Osaka Map Navi CSV 중 필요한 카테고리 일부

수집 제한:

- 문화·관광 시설 5개
- 명소·구적 5개
- 철도·버스 앵커 3개
- MCP 상품은 입장권/투어 중심으로 경험당 최대 5개만 연결

제외:

- 공원·스포츠 전체 수집
- 관공서/학교/주차장/방재 데이터
- 오사카 전체 시설 bulk import

### 후쿠오카

사용:

- BODIK/CKAN 후쿠오카 야타이 공개 데이터셋
- MyRealTrip MCP 상품 데이터

수집 제한:

- 야타이 6개
- 대표 먹거리/야간 경험 3개
- 역/숙소/공항 앵커 3개

제외:

- 신청형 실시간 개점 API
- 30분 단위 영업 상태
- 후쿠오카 전체 공공데이터 탐색

### 부산

사용:

- MyRealTrip MCP 상품 데이터
- 부산 맛집 데이터는 2차. 1차에서는 seed 또는 공개 문서 기반 소량만 사용

수집 제한:

- 해안/야경/시장/미식 중심 경험 8~10개
- 맛집/카페 공식 자산 최대 6개

제외:

- 부산 전체 음식점 bulk import
- 리뷰/사진 원문 저장
- 실시간 인구/상권 데이터

## 4. 1차 MVP에서 보류하는 소스

다음 소스는 중요하지만 1차 데모에서는 보류한다.

- TourAPI 전체 도시 자동 수집
- 공공데이터포털 관광식당 전국 수집
- 서울 열린데이터광장
- 스마트서울맵
- 도쿄 관광 통계 CSV
- 대구 맛집 전체 API
- BODIK generic connector 전체 탐색

보류 이유:

- 데이터가 많아지면 MVP 화면이 무거워진다.
- API 키/신청/인코딩/스키마 차이에 시간을 많이 쓰게 된다.
- 현재 목표는 데이터 플랫폼 완성이 아니라 제품 구조와 사업개발 사고를 보여주는 것이다.

## 5. 화면 노출 기준

사용자 화면:

- 한 도시에서 최대 12개 경험만 노출
- 경험 카드는 `대표 경험`, `숨은 선택지`, `먹거리`, `이동/숙소 앵커`로 묶는다.
- 상품은 카드 안에서 바로 쏟아내지 않고, 사용자가 경험을 고른 뒤 구매방식 비교 단계에서 보여준다.

내부 리포트:

- 모든 원천 데이터를 보여주지 않는다.
- 수집된 공식 자산 수, 상품 연결 수, 공급 부족 후보 수, 비용 추정 범위만 집계한다.
