- 검증 대상: `submission/src`

## 검증 범위 선언

개인 개발과 제출 일정을 고려해 실제 사용자 대상 테스트는 수행하지 않았다. 아래 결과는 기능 정확성과 실패 처리를 확인한 자체 테스트이며, 사용자 행동이나 사업 효과의 검증 결과가 아니다.

## 단위 테스트

```text
Ran 17 tests
OK
```

검증한 규칙:

1. HTML 포함 상세 설명을 근거용 일반 텍스트로 변환
2. 요청 도시 불일치 상품 제외
3. 여러 검색 페이지 순회 및 `gid` 중복 제거
4. 페이지 제한 시 불완전 조회 상태 표시
5. 상세 근거와 공식 상품 URL 보존
6. 옵션 목록이 비어 있으면 선택 날짜에 예약 불가로 판정
7. 상세 조회 실패를 기록하면서 검색 상품과 공식 URL은 보존
8. HTTPS와 마이리얼트립 공식 서브도메인만 허용
9. 접근 가능한 공식 리디렉션 허용
10. 외부 도메인으로 향하는 리디렉션 차단
11. 외부 리디렉션을 따라가기 전에 요청 자체를 차단
12. 입장권을 경험 키워드가 아닌 구매 방식으로 분류
13. 관광형 이동수단과 순수 이동 편의를 구분
14. 공식 카테고리를 각각 조회하고 인기순 정렬을 요청하지 않음
15. `전체`·유심·여행용품·순수 편의 카테고리는 건너뛰고 이동·교통은 유지
16. 투어는 발견 시 상세 조회, 입장권은 제목 우선, 패스는 선택 후 상세 조회로 분기
17. 카테고리별 수집은 기본적으로 마지막 페이지까지 진행하며 임의의 상품 수 제한을 두지 않음

## 실제 URL 스모크 테스트

- 대상: `https://experiences.myrealtrip.com/products/3887275`
- 최종 URL: 동일한 마이리얼트립 공식 상품 URL
- HTTP 상태: `200`
- 판정: `reachable=true`

이 검사는 테스트 시점의 접근 가능성만 보장한다. 실제 사용자 클릭 시점의 판매 상태·재고·가격은 상품 페이지에서 다시 확인해야 한다.

검색 정렬값을 생략하면 MCP 서버 기본값 `recommended`가 적용된다. 따라서 카테고리별 수집으로 단일 인기 목록 편중은 줄이지만, 서버 측 순위 편향이 완전히 제거됐다고 주장하지 않는다.

## 경쟁 서비스 대비 검증한 차별점

기존 여행 서비스와 AI 여행 플래너는 보통 인기 상품, 상품 유형, 가격, 일정 또는 개인화 추천을 먼저 보여준다. 이번 플러그인은 상품을 즉시 추천하지 않고 도시별 경험 지도를 먼저 보여준 뒤, 사용자가 고른 경험에 대해서만 구매 방식을 펼치는 구조로 설계했다. 이 구조가 실제 탐색 피로를 줄이는지는 사용자 테스트가 필요하지만, 문서·스킬·테스트 기준에서는 다음 차이를 확인했다.

- 첫 화면은 상품 목록이 아니라 겹치지 않는 경험의 `전체 보기`다.
- `입장권·패스·투어`는 경험 키워드가 아니라 선택 뒤 나타나는 구매 방식이다.
- `마리오 카트: 쿠파의 도전장`은 USJ 입장권, 슈퍼 닌텐도 월드 확약·정리권 관련 상품, 익스프레스 패스처럼 실제 이용 방식이 명시된 상품만 연결한다.
- `아베노 하루카스 300`은 단독 입장권을 중심으로, 패스 포함 여부와 하루카스 방문이 명시된 투어만 대체 이용 방식으로 연결한다.
- 여러 상품과 후기 근거가 반복되는 경험은 안정적 콘텐츠 후보로, 상품 근거는 있지만 반복이 적은 경험은 신규 발굴 후보로 분리한다.

다만 상품 겹침 수는 마이리얼트립 내 노출·검증 정도이지 실제 방문객 수나 한국인 밀도를 뜻하지 않는다.

- 오사카: 슈퍼 닌텐도 월드, 마리오 카트, 오코노미야키, 모던야키
- 부산: 광안대교 야경 요트, 스카이캡슐, 감천문화마을
- 런던: 해리포터, 아스널, 토트넘, 위키드
- 충칭: 리즈바역, 래플즈시티, 충칭 소면
- 전주: 한옥 스냅, 경기전, 전라감영
- 삿포로: 청의 호수, 흰수염폭포, 준페이 식사, 샤부샤부

## 잔여 위험

- MCP 호출 제한·응답 지연으로 상품이 많은 도시의 전체 페이지 수집이 중단되면 롱테일 콘텐츠가 누락될 수 있다.
- 도시 별칭과 지역명 차이로 정확 도시 필터가 거짓 음성을 만들 수 있다.
- 키워드 추출은 모델 판단을 포함하므로 완전히 결정적이지 않다.
- 먹거리 예약 상품이 적은 도시에서는 결과가 비거나 관광 상품 속 포함 식사에 편중될 수 있다.
- URL 검사는 일시 장애·봇 차단을 실제 링크 장애로 오인할 수 있다.
- 자체 테스트는 사용자 발견 경험과 결제 효과를 증명하지 않는다.

위 사용자 가치 지표는 제출 이후 후속 검증 대상으로 유지한다.

일반 사용자에게 Partner API 키를 요구하지 않는다. REST 개발자는 `MYREALTRIP_API_KEY` 환경변수만 사용하며 대화나 로그에 남기지 않는다.
