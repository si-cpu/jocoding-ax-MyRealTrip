# MRT Idea Gate

개인적인 아이디어 한 건을 검증 가능한 명제로 분해하고, 공개 근거와 반례를 대조하여 정식 기획에 들어갈 초기 가치만 판단한다. 아이디어를 그럴듯하게 포장하거나 최종 승인하지 않는다.

## 시작 조건 확인

다음 범위에 해당할 때만 진행한다.

- 아직 정식 기획으로 채택되지 않은 아이디어 한 건이다.
- 사용자는 서비스 기획자 또는 콘텐츠 기획자의 관점이다.
- 목표는 PRD 작성 이전의 최초 확인이다.

이미 정식 기획이 시작됐다면 이 스킬의 범위를 짧게 알리고, 사용자가 원하면 아이디어 단계의 핵심 가정만 역으로 추출해 검토한다. 상세 요구사항, 일정, 예산, 화면, 프로덕션 코드 또는 최종 승인 판단으로 확장하지 않는다.

## 최소 입력 수집

다음 세 항목을 확보한다.

1. 역할: 서비스 기획 또는 콘텐츠 기획
2. 아이디어 한 문장
3. 아이디어가 떠오른 경험이나 관찰

일부가 빠졌으면 한 번에 필요한 항목만 짧게 질문한다. 사용자가 충분한 서술을 제공했다면 형식적인 재질문 없이 추출하여 진행한다. 자료나 수치가 없어도 시작하되, 없는 근거를 보충하지 않는다.

## 검토 절차

### 1. 원문을 보존한다

사용자의 아이디어와 경험을 의미가 달라지지 않도록 그대로 인용한다. 맞춤법을 고친 문장을 원문처럼 제시하지 않는다.

### 2. 문제와 해결책을 분리한다

다음 항목을 추출한다.

- 예상 대상 사용자
- 문제가 발생하는 상황
- 관찰한 문제 또는 기회
- 제안한 해결 방향
- 기대 변화
- 아직 확인되지 않은 가정

사용자가 말하지 않은 대상, 빈도, 원인, 수치 또는 사업 효과를 추가하지 않는다.

### 3. 아이디어를 원자 명제로 분해한다

한 문장에 하나의 주장만 두고, 반대 증거가 나오면 틀렸다고 판단할 수 있게 쓴다. 필요한 유형만 사용한다.

- 대상: 문제를 경험하는 구체적인 사용자가 존재한다.
- 상황: 문제가 발생하는 시점과 조건이 존재한다.
- 문제: 관찰한 불편 또는 기회가 실제 현상이다.
- 중요성: 반복성이나 영향이 정식 기획을 검토할 수준이다.
- 기존 대안: 현재 기능이나 우회 방법으로 충분히 해결되지 않는다.
- 방향: 제안한 해결 방향이 문제와 논리적으로 연결된다.
- 기업 적합성: 마이리얼트립이 다룰 이유와 맥락이 있다.

각 명제마다 틀렸다고 판단할 수 있는 조건을 함께 쓴다. 상세 기준은 [judgment-rules.md](references/judgment-rules.md)를 따른다.

### 4. 사실, 경험, 추론과 가정을 분리한다

사용자가 제공한 자료와 공개적으로 검증 가능한 자료만 사용한다. 웹 접근이 가능하면 중요한 핵심 명제를 먼저 검색하고, 마이리얼트립 공식 페이지·도움말·보도자료처럼 직접적인 출처를 우선한다. 검색 결과의 요약문만으로 단정하지 말고 원문을 확인한다.

수집한 내용을 다음 네 영역에 서로 섞지 않고 기록한다.

- `FACT`: 원문 출처 또는 제공된 데이터에서 직접 확인한 사실. 반드시 출처 ID를 붙인다.
- `EXPERIENCE`: 사용자가 직접 겪거나 관찰했다고 말한 사례. 출처는 사용자 진술이며 일반화하지 않는다.
- `INFERENCE`: 하나 이상의 사실·경험에서 도출한 해석. 사용한 항목 ID와 추론 과정을 적는다.
- `ASSUMPTION`: 아이디어 성립에 필요하지만 아직 확인하지 못한 주장. 사실처럼 서술하지 않는다.

`UNKNOWN`은 위 분류가 아니라 명제를 판정할 근거가 부족한 상태로만 사용한다. 출처가 없는 숫자와 시장·매출 효과를 만들지 않는다. 개인 경험 하나로 빈도, 전체 사용자 비율, 원인 또는 사업 효과를 일반화하지 않는다.

### 5. 출처를 명제까지 추적 가능하게 기록한다

외부 사실마다 `S1`, `S2`처럼 출처 ID를 부여하고 [source-rules.md](references/source-rules.md)에 따라 출처 원장을 작성한다. 각 사실에는 출처 ID를, 각 명제에는 사용한 사실·경험·추론·가정 ID를 연결한다.

원문을 직접 열지 못했거나 검색 결과 요약만 확인한 자료는 `SEARCH_LEAD`로 표시한다. `SEARCH_LEAD`는 추가 확인 후보일 뿐 `FACT`나 최종 권고의 근거로 사용하지 않는다.

### 6. 반례와 더 작은 대안을 찾는다

아이디어가 가치 없을 수 있는 가장 강한 이유를 최소 하나 제시한다. 단순한 반대 의견이 아니라, 사실이라면 현재 권고를 바꿀 조건이어야 한다.

다음을 우선 확인한다.

- 문제가 드물거나 영향이 작을 가능성
- 기존 기능, 외부 도구 또는 사용자 우회 행동으로 충분한 가능성
- 제안 기능 없이 문구·정보 구조·운영 방식으로 해결할 가능성
- 콘텐츠가 기존 테마와 실질적으로 중복될 가능성
- 마이리얼트립보다 다른 주체가 해결하기 적합할 가능성

반례를 확인할 현재 상태와 가장 작은 확인 행동도 함께 적는다.

### 7. 역할별 초기 점검을 한다

서비스 기획 아이디어에서는 다음만 점검한다.

- 문제 발생 사용자와 사용자 여정이 구체적인가
- 현재 기능 또는 우회 행동이 무엇인가
- 기능 추가보다 작은 해결책이 있는가
- 치명적인 예외·오작동·운영 위험이 보이는가
- 기대 변화가 관찰 가능한가

콘텐츠 기획 아이디어에서는 다음만 점검한다.

- 대상 여행자와 소비 상황이 구체적인가
- 기존 콘텐츠·테마·프로모션과 어떻게 다른가
- 정보 출처, 최신성, 정확성 또는 저작권 위험이 있는가
- 시즌·지역·상품 공급 조건과 연결 가능한가
- 노출 뒤 기대 행동이 관찰 가능한가

정보가 없으면 부적합으로 단정하지 말고 `UNKNOWN`으로 표시한다.

### 8. 명제 상태와 아이디어 권고를 정한다

각 명제를 `SUPPORTED`, `PARTIAL`, `CONTRADICTED`, `UNKNOWN` 중 하나로 판정한다. 명제 상태를 점수화하거나 단순 평균하지 않는다.

- `GO`: 대상·상황·문제의 존재가 지지되고, 중요성과 기업 적합성이 최소 부분 지지되며, 현재 치명적 반례가 없다.
- `REVISE`: 문제에는 가치가 있지만 대상이나 해결 방향이 어긋났거나 더 작은 대안이 더 적합하다.
- `HOLD`: 문제·중요성처럼 결론을 좌우하는 핵심 명제가 미확인이고 작은 검증 행동이 먼저 필요하다.
- `DROP`: 핵심 문제가 직접 반박됐거나 기존 대안이 충분하거나 기업 적합성이 현저히 낮다는 근거가 있다.

근거를 찾지 못했다는 사실만으로 `CONTRADICTED` 또는 `DROP`을 선택하지 않는다. 해결 방향이 약하다는 이유만으로 실제 문제까지 부정하지 않는다.

### 9. 가장 작은 다음 행동을 제안한다

가장 약한 핵심 명제 하나를 고르고, 판정을 바꾸는 데 필요한 최소 행동 하나를 제시한다. 예시는 공개 문의·리뷰 표본 확인, 기존 흐름 재현, 5건 내외의 탐색 인터뷰, 소규모 콘텐츠 중복 점검 등이다. 표본 수를 보편 법칙처럼 제시하지 말고 확인 목적과 중단 조건을 설명한다.

## 출력

[output-template.md](references/output-template.md)의 순서로 한국어 검토 카드 한 장을 작성한다. 사용자가 별도 형식을 요청하면 항목은 유지하되 표현 형식만 바꾼다.

최종 권고 뒤에는 반드시 다음 문장을 포함한다.

> 이 결과는 현재 확인된 근거에 따른 초기 권고이며, 사람을 대신한 최종 승인 또는 중단 결정이 아닙니다.
