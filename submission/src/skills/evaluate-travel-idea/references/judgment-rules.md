# 명제 및 근거 판단 규칙

## 좋은 명제의 조건

명제는 다음 조건을 모두 만족해야 한다.

- 하나의 주장만 담는다.
- 대상, 상황, 현상 또는 기대 결과가 드러난다.
- 공개 자료, 제공 데이터 또는 사용자 조사로 확인 가능하다.
- 틀렸다고 볼 조건을 미리 적을 수 있다.
- 근거에 없는 수량, 빈도, 인과관계를 포함하지 않는다.

나쁜 예:

> 친구 여행은 결제가 불편하고 분할결제를 만들면 만족도와 매출이 높아질 것이다.

분해 예:

- P1. 여러 명이 하나의 상품을 예약할 때 대표자 한 명이 전체 금액을 결제하는 상황이 존재한다.
- P2. 대표 결제자는 동행자와 비용을 정산하는 과정에서 불편을 경험할 수 있다.
- P3. 이 불편은 정식 기획을 검토할 만큼 반복되거나 영향이 크다.
- P4. 현재 송금·더치페이 방식만으로는 이 불편이 충분히 해결되지 않는다.
- P5. 서비스 내부 분할결제가 다른 대안보다 문제를 더 잘 해결할 가능성이 있다.

## 근거의 힘

강한 순서가 항상 고정되는 것은 아니지만 다음 원칙을 적용한다.

1. 해당 현상을 직접 측정한 원자료나 공식 기능 문서
2. 조사 방법과 범위가 공개된 연구·통계·보고서
3. 다수의 독립적인 공개 사례나 리뷰
4. 사용자의 직접 경험
5. 사실을 연결한 추론
6. 출처 없는 주장

공식 자료는 기능의 존재를 확인하는 데 강하지만 사용자 불편의 크기를 자동으로 증명하지 않는다. 리뷰는 문제 사례를 보여줄 수 있지만 전체 사용자 빈도를 자동으로 대표하지 않는다.

## 주장 성격과 명제 상태를 분리한다

주장 성격과 명제 상태는 서로 다른 축이다.

| 주장 성격 | 의미 | 필수 연결 |
|---|---|---|
| `FACT` | 원문에서 직접 확인한 사실 | 출처 ID `S#` |
| `EXPERIENCE` | 사용자의 직접 경험 또는 관찰 | 경험 ID `E#` |
| `INFERENCE` | 사실·경험을 연결해 도출한 해석 | 전제 ID와 추론 과정 |
| `ASSUMPTION` | 아이디어 성립에 필요하지만 미확인인 주장 | 확인 방법과 실패 조건 |

예를 들어 `사용자가 여행 중 관심 콘텐츠를 자주 놓친다`는 사용자의 한 사례만 있을 때 `EXPERIENCE`이면서 명제 상태는 `PARTIAL`이다. `마이리얼트립이 특정 티켓을 판매한다`는 공식 상품 페이지로 확인하면 `FACT`이며, 해당 상품 존재 명제는 `SUPPORTED`일 수 있다.

`UNKNOWN`은 주장 성격이 아니다. 명제에 필요한 사실이 부족하다는 판정 상태다.

## 명제 상태

### SUPPORTED

- 명제의 핵심 범위를 직접 지지하는 근거가 있다.
- 신뢰할 만한 반대 근거가 없거나 결론을 바꿀 수준이 아니다.

### PARTIAL

- 일부 대상·상황만 확인됐다.
- 간접 근거 또는 단일 경험만 있다.
- 지지 근거와 반대 근거가 함께 존재한다.

### CONTRADICTED

- 같은 대상과 상황을 다룬 직접 근거가 명제와 반대된다.
- 이미 존재하는 기능이 ‘기존 대안이 없다’는 명제를 반박한다.

### UNKNOWN

- 판단할 자료가 없다.
- 자료는 있지만 대상·시점·상황이 달라 적용하기 어렵다.
- 검색하지 못했거나 접근할 수 없었다.

`검색 결과 없음`과 `존재하지 않음`을 구분한다.

## 충돌 처리

- 출처의 조사 대상, 시점, 정의와 방법을 비교한다.
- 최신 자료라는 이유만으로 조사 범위가 더 적합한 과거 자료를 폐기하지 않는다.
- 충돌을 숨기지 말고 양쪽 근거를 함께 기록한다.
- 해결되지 않은 충돌은 `PARTIAL` 또는 `UNKNOWN`으로 남긴다.

## 명제별 필수 기록

각 명제마다 다음을 남긴다.

```text
명제 ID:
명제:
유형:
틀렸다고 볼 조건:
지지 항목 ID(F#/E#/I#):
반대 항목 ID(F#/E#/I#):
미검증 가정 ID(A#):
상태:
판단 이유:
다음 확인 행동:
```

아이디어 전체 권고는 핵심 명제의 상태와 반례를 바탕으로 정하며, 명제 개수나 점수의 평균으로 계산하지 않는다.
