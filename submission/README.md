# 여행 경험 진열대 — Codex 플러그인 v0.6.0

목적지를 입력하면 마이리얼트립 상품을 카테고리별로 균형 있게 확인해, 입장권·패스·투어라는 판매 형식에 가려진 구체적인 여행 경험을 진열대처럼 펼쳐주는 플러그인입니다. 경험을 고른 다음에야 이용 가능한 구매 방식과 공식 상품을 보여줍니다.

## 해결하려는 문제

목적지를 정한 여행자는 그곳에서 무엇을 하고 먹을지 구체적으로 떠올리지 못해 긴 상품 목록을 반복해서 살펴보게 됩니다. 인기 상품과 `투어·입장권·패스` 중심의 진열은 판매 방식을 알려주지만 그 안의 낯선 경험을 가릴 수 있습니다. 이 플러그인은 `슈퍼 닌텐도 월드·고베 누노비키 허브원·아스널·오코노미야키`처럼 여행 목적이 그려지는 단위로 상품을 다시 색인합니다.

이 플러그인은 사용자를 대신해 최적 상품을 추천하지 않습니다. 가능한 경험을 먼저 펼치고 사용자가 관심 키워드를 직접 선택하게 합니다.

## 사용법

여행지만 입력하면 됩니다. 날짜는 선택사항입니다.

```text
오사카에서 뭐 하고 뭐 먹을 수 있어?
런던 상품을 해리포터·축구·공연 같은 키워드로 보여줘.
2026-08-10 부산에서 예약 가능한 경험을 찾아줘.
```

첫 결과는 다음처럼 경험 중심으로 표시됩니다.

```text
오사카에서 발견할 수 있는 경험

자연·근교
- 고베 누노비키 허브원

테마파크·작품
- 유니버설 스튜디오 재팬
- 슈퍼 닌텐도 월드

오사카에서 뭐 먹지?
- 오코노미야키
- 타코야키
- 모던야키
```

사용자가 경험을 선택하면 `단독 입장권·패스 포함·투어 포함·관광형 이동수단`처럼 그 경험을 이용하는 방식을 나누고, 직접 근거가 있는 공식 상품 링크를 펼칩니다.

## 작동 절차

1. 공식 카테고리 API로 해당 도시의 상품 카테고리를 가져옵니다.
2. 각 카테고리를 별도로 조회하며 판매량·평점·가격 정렬을 요청하지 않습니다.
3. 검색 응답의 `도시 ∙ 카테고리`로 다른 도시 상품을 제외하고, 카테고리 간 `gid` 중복을 제거합니다.
4. 상품 상세의 제목, 소개, 포함사항, 일정 제목·설명을 확인합니다.
5. `경험 자체`, `그 경험의 구매 방식`, `순수 이동 편의`를 서로 다른 축으로 판정합니다.
6. 장소·시설·작품·팀·공연·음식·구체적인 행동은 경험으로, 입장권·패스·투어는 구매 방식으로 저장합니다.
7. 로프웨이·크루즈·관광열차처럼 그 자체가 볼거리인 이동은 경험으로 보존하고, 공항 픽업·유심·보험 같은 순수 편의는 제외합니다.
8. 경험을 채워진 테마별로 펼치고 이름순으로 안정적으로 표시합니다. 근거 상품 수를 인기 순위처럼 사용하지 않습니다.
9. 사용자가 경험을 고르면 구매 방식별 공식 상품을 연결하고, 노출 직전 URL 안전성을 확인합니다.
10. 날짜가 입력되면 옵션 API로 그 날짜의 옵션 존재 여부를 확인합니다.

## 판단 기준

- 상품명·소개·일정에 직접 등장하거나 검증된 공식 별칭일 때만 키워드와 상품을 연결합니다.
- `애니메이션 관련 상품`이라는 이유만으로 특정 작품인 `나루토`를 생성하지 않습니다.
- 하나의 상품에서 여러 키워드를 꺼낼 수 있지만, 각 키워드는 독립된 원문 근거를 가져야 합니다.
- `오코노미야키`와 `모던야키`, `해변열차`와 `스카이캡슐`처럼 다른 경험을 과도하게 합치지 않습니다.
- 음식은 식사권, 식당 예약, 푸드투어, 쿠킹클래스 또는 포함 식사처럼 상품과 직접 연결되는 경우를 우선합니다.
- 전체 페이지나 상세 조회가 제한되면 `이번에 확인한 상품 범위에서`라고 표시합니다.
- 사용자의 성향을 추론해 `나다움`을 결정하지 않습니다. 다양한 경험을 보여주고 무엇을 고르는지는 여행자에게 남깁니다.
- 입장권·패스·투어는 키워드가 아니라 선택한 경험을 예약하는 방식입니다.

세부 규칙은 `skills/map-travel-content/references/content-indexing-rules.md`에 있습니다.

## 정보 부족과 실패 처리

- API 키가 없고 공식 MCP도 사용할 수 없으면 일반 웹 정보로 결과를 대체하지 않습니다.
- 상품 상세가 없으면 검색 상품명에서 직접 확인되는 키워드만 사용합니다.
- 선택 날짜의 옵션이 비어 있으면 해당 날짜에 예약 가능하다고 표현하지 않습니다.
- 먹거리 키워드는 유명하더라도 마이리얼트립 상품 근거가 없으면 상품 링크를 만들지 않습니다.
- 상품 URL이 공식 도메인이 아니거나 접근 실패·외부 리디렉션이 발생하면 링크를 제외합니다.
- 예약·결제·취소는 수행하지 않으며 최종 가격과 재고는 상품 페이지에서 확인하게 합니다.

## 데이터 연결

- Partner REST API: `https://partner-ext-api.myrealtrip.com`
- 공식 MCP: `https://mcp-servers.myrealtrip.com/mcp`
- REST 인증: `MYREALTRIP_API_KEY` 환경변수

API 키를 프롬프트, 코드, `.mcp.json`, 테스트 또는 제출 로그에 입력하지 마세요.

## 직접 시험하기

### 1. 데이터 수집기만 확인

프로젝트 루트에서 API 키를 셸 환경변수로 설정한 뒤 실행합니다. 키 값은 대화창이나 명령 기록에 붙여넣지 마세요.

```bash
python3 submission/src/scripts/myrealtrip_api.py tna-shelves --city 오사카 --pretty
```

결과의 `shelves`에는 공식 카테고리별 상품과 `purchaseForm`이 나타나야 하며, 요청 payload에는 판매량 정렬이 포함되지 않습니다. 키가 없으면 비밀정보를 요구하지 않고 `MYREALTRIP_API_KEY is not set` 오류로 중단하는 것이 정상입니다.

### 2. Codex 개인 플러그인으로 설치

처음 한 번만 개인 플러그인과 마켓 항목을 만든 뒤 이 제출물의 `src`를 복사합니다.

```bash
python3 ~/.codex/skills/.system/plugin-creator/scripts/create_basic_plugin.py travel-content-map --with-marketplace --with-skills --with-mcp
cp -R submission/src/. ~/plugins/travel-content-map/
codex plugin add travel-content-map@personal
```

설치 후에는 변경 내용을 확실히 불러오도록 새 Codex 대화를 시작합니다. 다음 순서로 시험합니다.

```text
오사카에서 발견할 수 있는 경험을 보여줘.
자연·근교 경험을 더 펼쳐줘.
고베 누노비키 허브원을 선택할게. 이용 방법을 입장권·패스·투어로 나눠줘.
```

첫 답변에 입장권·패스·투어가 경험 키워드처럼 섞이지 않고, 경험 선택 뒤 구매 방식으로 나타나는지 확인합니다. 이미 개인 플러그인이 있다면 파일을 다시 복사한 뒤 다음처럼 캐시를 갱신하고 새 대화에서 확인합니다.

```bash
python3 ~/.codex/skills/.system/plugin-creator/scripts/update_plugin_cachebuster.py ~/plugins/travel-content-map
codex plugin add travel-content-map@personal
```

## 현재 검증 범위

- 페이지 순회 및 `gid` 중복 제거
- 요청 도시 불일치 상품 제거
- 카테고리별 균형 수집과 인기순 미사용
- 관광형 이동과 순수 이동 편의 구분
- 경험과 구매 방식 분리
- HTML이 포함된 상세 설명의 텍스트 정리
- 상세 근거와 공식 URL 보존
- 날짜 옵션이 없는 상품을 예약 가능으로 표시하지 않는 로직
- 비공식 URL과 외부 도메인 리디렉션 차단
- 실제 마이리얼트립 상품 URL의 HTTP 200 응답 확인
- 오사카·부산·런던·충칭·전주·삿포로 공개 상품 표본의 콘텐츠 추출 가능성

이번 제출은 개발자 자체 테스트만 수행합니다. 실제 여행자를 대상으로 한 비교 실험은 일정상 후속 검증으로 남기며, 탐색 시간·상품 클릭·결제 전환 개선을 입증된 효과로 주장하지 않습니다.

실제 Partner REST 호출은 로컬 환경에 API 키를 설정한 뒤 추가 검증해야 합니다.

## 향후 운영 계획

MVP의 데이터 한계를 줄이기 위한 운영 고도화 계획은 [FUTURE_OPERATIONS.md](FUTURE_OPERATIONS.md)에 정리했습니다.

- 전체 상품 사전 색인으로 롱테일 누락 완화
- 도시 ID·별칭 사전으로 `충칭/중경` 같은 표기 통합
- 근거 기반 콘텐츠 표준 사전으로 키워드 표현 통제
- 관광 공공데이터로 정보성 먹거리 보완
- 운영 이후 퍼널 분석과 A/B 테스트로 사용자 효과 검증

외부 먹거리 데이터는 마이리얼트립 예약 상품과 분리해 출처를 표시하며, 현재 MVP에 구현된 기능으로 표현하지 않습니다.

## 제한사항

- 상품 설명의 품질과 범위에 따라 키워드 수가 달라집니다.
- 기본값은 카테고리별 검색 1페이지이므로 상품이 많은 도시는 전체가 아닌 `이번에 카테고리별로 확인한 상품 범위`만 보여줄 수 있습니다.
- 도시 표기가 `충칭/중경`처럼 다르면 정확 일치 필터가 관련 상품까지 누락할 수 있습니다.
- 구체적인 키워드 추출과 동의어 판단은 Codex가 수행하므로 같은 원문에서도 표현이 조금 달라질 수 있습니다.
- 먹거리 예약 상품이 적은 도시에서는 `뭐 먹지?` 결과가 비어 있을 수 있습니다.
- 정상 상품 페이지도 일시 장애·접속 제한·봇 차단으로 URL 검사에서 제외될 수 있습니다.
- 마이리얼트립에서 판매하지 않는 현지 경험은 유명하더라도 결과에 나타나지 않습니다.
- 키워드 탐색이 결제 전환을 높인다는 효과는 아직 입증하지 않았습니다.
- 실제 Partner REST API 키 환경의 전체 흐름은 아직 검증하지 않았습니다.
- 개인화, 예약 이력 조회, 항공·숙소, 자동 일정 생성은 MVP 범위가 아닙니다.
