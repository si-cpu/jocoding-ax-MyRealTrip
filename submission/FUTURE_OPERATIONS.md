# 향후 운영 기획: 여행 콘텐츠 지도 고도화

## 1. 문서 목적

현재 MVP는 마이리얼트립 상품 API를 실시간 조회해 `뭐 하지?`와 `뭐 먹지?` 키워드를 만드는 구조다. 이 문서는 MVP에서 확인된 데이터 한계를 운영 단계에서 어떻게 줄일지 제시한다.

다음 기능은 **현재 플러그인에 구현됐다고 주장하지 않는다.** 마이리얼트립과 운영 환경이 확보된 이후 단계적으로 적용할 계획이다.

## 2. 해결 과제와 운영 방향

| 과제 | 원인 | 운영 해결책 | 해결 가능 수준 |
|---|---|---|---|
| 롱테일 상품 누락 | 대화마다 검색 3페이지·상세 100개로 제한 | 전체 상품 사전 색인과 증분 갱신 | 높음 |
| 도시 별칭 누락 | TNA 검색 결과의 도시명 정확 일치 | 표준 도시 ID와 별칭 사전 | 높음 |
| 키워드 표현 차이 | 자유 서술형 상품 설명과 모델 판단 | 표준 콘텐츠 사전과 근거 기반 승인 | 중~높음 |
| 먹거리 상품 부족 | 마이리얼트립에 예약 상품 자체가 적은 도시 존재 | 공공 관광 데이터로 정보 영역 보완 | 부분 가능 |
| 사용자 효과 미검증 | 실제 사용자 행동 데이터 부재 | 단계적 출시 후 퍼널 분석·A/B 테스트 | 운영 이후 가능 |

## 3. 목표 운영 구조

```text
마이리얼트립 상품 API
        │
        ▼
전체 상품 수집·상세 갱신 ── 실패·변경 감시
        │
        ▼
도시 표준화 ── 도시 ID·한글/영문/현지어 별칭
        │
        ▼
콘텐츠 추출 ── AI 후보 생성 + 직접 근거 보존
        │
        ▼
콘텐츠 표준화 ── 작품·장소·팀·공연·음식 사전
        │
        ├── 예약 가능 콘텐츠: 마이리얼트립 공식 상품 연결
        └── 정보성 먹거리: 공공 출처 표시, 예약 가능 표현 금지
```

사용자 요청 시에는 상품 API 전체를 다시 읽지 않고 사전 생성된 색인을 조회한다. 최종 상품 링크와 날짜 옵션만 실시간으로 재검증한다.

## 4. 단계별 운영 계획

### Phase 1. 전체 상품 사전 색인

목표: 현재 MVP의 페이지·상세 조회 제한으로 생기는 롱테일 누락을 줄인다.

운영 방식:

1. TNA 검색의 `hasNextPage`가 끝날 때까지 도시별 전체 페이지를 순회한다.
2. `gid`로 중복을 제거하고 상품 상세를 수집한다.
3. 상품명, 설명, 포함사항, 일정, 공식 URL과 수집 시각을 저장한다.
4. 실패한 상세 조회는 재시도 큐에 넣고 검색 제목 근거만 임시 사용한다.
5. 정기 전체 갱신과 링크·옵션 실시간 확인을 분리한다.

권장 주기:

- 상품 목록·상세: 매일 1회 또는 API 정책에 맞춘 주기
- 실패 상품 재시도: 지수 백오프 적용
- 사용자에게 노출할 상품 URL: 노출 직전 확인
- 날짜별 예약 옵션: 사용자가 날짜를 입력했을 때 확인

품질 기준:

- 모든 페이지 순회가 끝난 도시만 `전체 상품에서`라고 표시한다.
- 일부 호출이 실패하면 `확인된 상품 범위에서`라고 표시한다.
- 삭제·판매 중지 상품은 색인에서 비활성화하되 이력을 남긴다.

### Phase 2. 도시 표준 사전

목표: `충칭·중경·Chongqing`처럼 같은 도시의 다른 표현 때문에 발생하는 누락을 줄인다.

최소 데이터 구조:

```json
{
  "canonicalCityId": "cn-chongqing",
  "displayName": "충칭",
  "countryCode": "CN",
  "aliases": ["충칭", "중경", "Chongqing", "重庆"],
  "source": "OFFICIAL_OR_REVIEWED",
  "status": "ACTIVE"
}
```

운영 원칙:

- 마이리얼트립의 구조화된 지역 식별자를 사용할 수 있으면 우선한다.
- 구조화된 식별자가 없으면 공식 표기와 운영자 검토로 별칭을 등록한다.
- 같은 이름의 다른 도시가 존재할 수 있으므로 국가 코드와 함께 판정한다.
- `연해주 → 블라디보스토크·우수리스크`처럼 지역을 여러 도시로 확장할 때 사용자에게 범위를 표시한다.
- 자동 생성된 별칭은 검토 전 상품 연결에 사용하지 않는다.

### Phase 3. 콘텐츠 표준 사전

목표: 모델이 생성하는 키워드의 표기 차이와 과도한 병합을 줄인다.

최소 데이터 구조:

```json
{
  "contentId": "jp-osaka-usj",
  "canonicalLabel": "유니버설 스튜디오 재팬",
  "type": "DO",
  "aliases": ["USJ", "유니버셜 스튜디오 재팬"],
  "cityIds": ["jp-osaka"],
  "evidencePolicy": "DIRECT_OR_APPROVED_ALIAS"
}
```

처리 절차:

1. AI는 상품 원문에서 콘텐츠 후보와 근거 문구를 추출한다.
2. 기존 사전과 일치하면 표준 키워드로 변환한다.
3. 새 후보는 빈도, 연결 상품 수, 근거 강도를 계산한다.
4. 모호한 후보는 검토 큐로 보내고 사용자에게 바로 노출하지 않는다.
5. 승인된 별칭과 병합 규칙만 운영 사전에 반영한다.

자동 승인 가능 조건:

- 상품 제목이나 일정 제목에 직접 등장
- 기존 표준 키워드의 승인된 별칭과 정확히 일치
- 도시와 콘텐츠 유형이 충돌하지 않음

검토가 필요한 조건:

- 상품 설명에만 한 번 등장
- 특정 작품·음식으로 추론해야 함
- 서로 다른 경험을 하나로 합칠 가능성이 있음

### Phase 4. 먹거리 보조 데이터

목표: 마이리얼트립의 식사권·푸드투어가 적은 도시에서도 사용자가 대표 먹거리를 발견하게 한다.

허용 데이터:

- 국가·지역 관광청
- 지자체 관광 포털
- 공공 관광 데이터 API
- 출처와 갱신일을 확인할 수 있는 공개 데이터

출력은 반드시 두 영역으로 분리한다.

```text
예약 가능한 먹거리
- 마이리얼트립 상품과 공식 링크

현지 먹거리 정보
- 공공데이터에서 확인한 음식명과 출처
- 예약 가능 또는 마이리얼트립 상품이라고 표현하지 않음
```

운영 규칙:

- 마이리얼트립 상품 여부는 마이리얼트립 API만 기준으로 한다.
- 외부 데이터는 대표 음식 발견을 보완할 뿐 상품 재고·가격을 만들지 않는다.
- 출처 URL, 제공기관, 최종 갱신일을 저장한다.
- 출처가 사라지거나 갱신일을 확인할 수 없으면 비활성화한다.
- 식품 알레르기·건강 적합성을 추론하거나 보장하지 않는다.

### Phase 5. 사용자 효과 검증

목표: 키워드 탐색이 기존 상품 유형 탐색보다 실제로 도움이 되는지 확인한다.

검증 퍼널:

```text
도시 입력
→ 키워드 노출
→ 키워드 선택
→ 상품 상세 이동
→ 예약 화면 진입
→ 결제 완료
```

1차 지표:

- 첫 키워드 선택까지 걸린 시간
- 키워드 선택률
- 상품 상세 이동률
- 잘못 연결됐다고 신고된 비율

2차 지표:

- 예약 화면 진입률
- 결제 전환율
- 재방문 시 다른 키워드 선택률

검증 원칙:

- 초기에는 기존 탐색과 키워드 탐색을 비교한다.
- 결제 전환만 보지 않고 탐색 시간과 오인 연결을 함께 본다.
- 사용자 동의 없이 예약 이력을 개인화에 사용하지 않는다.
- 충분한 표본이 쌓이기 전에는 효과를 확정적으로 주장하지 않는다.

## 5. 운영 역할

| 역할 | 책임 |
|---|---|
| Product | 핵심 지표, 노출 정책, 실험 중단 기준 결정 |
| Data/Backend | 전체 색인, 갱신, 캐시, 실패 재처리 운영 |
| Content Operations | 도시·콘텐츠 별칭 검토와 오인 연결 수정 |
| Partner/API | 상품 필드·호출 정책 변경 공유 |
| Privacy/Security | 행동 데이터 최소 수집과 보관 정책 검토 |

개인 개발 단계에서는 이 역할을 시스템 규칙과 수동 검토로 축소해 운영하고, 실제 서비스 전환 시 책임을 분리한다.

## 6. 모니터링 지표

| 영역 | 지표 | 경고 조건 예시 |
|---|---|---|
| 수집 | 도시별 페이지 완료율 | 완료되지 않은 도시 증가 |
| 상세 | 상세 조회 성공률 | 연속 실패 또는 급격한 하락 |
| 링크 | 공식 URL 도달 성공률 | 4xx·5xx·외부 리디렉션 증가 |
| 도시 | 별칭 미매칭률 | 동일 도시 재검색 증가 |
| 콘텐츠 | 키워드당 근거 상품 수 | 근거 1개 키워드 비중 급증 |
| 품질 | 오인 연결 신고율 | 사전 기준 초과 |
| 먹거리 | 예약 가능/정보성 표시 오류 | 출처 혼동 1건 이상 |

정확한 임계값은 실제 트래픽을 관찰한 뒤 확정한다. 근거 없이 임의의 목표 수치를 제시하지 않는다.

## 7. 출시·중단 기준

단계적 출시 조건:

- 도시·상품·링크 근거를 추적할 수 있음
- 불완전 조회를 완전 조회로 표현하지 않음
- 외부 먹거리와 마이리얼트립 예약 상품을 명확히 구분
- API 키와 사용자 행동 데이터가 로그에 노출되지 않음

노출 중단 조건:

- 다른 도시 상품 연결이 반복됨
- 공식 상품이 아닌 링크가 노출됨
- 외부 정보를 마이리얼트립 예약 상품으로 오인하게 만듦
- 상품 API 변경으로 근거 필드를 신뢰할 수 없음

## 8. 최종 우선순위

1. 전체 상품 사전 색인
2. 도시 표준 사전
3. 콘텐츠 표준 사전
4. 먹거리 공개 데이터 보완
5. 사용자 효과 검증과 동의 기반 개인화

이 순서는 데이터 정확성을 먼저 확보한 뒤 탐색 범위와 개인화를 넓히기 위한 것이다. 현재 MVP의 핵심인 **공식 상품 근거와 사용자 직접 선택**은 운영 단계에서도 유지한다.
