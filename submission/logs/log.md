## Codex (진행 메시지)

좋아, 마이리얼트립 단일 기업으로 잡고 가자. 먼저 현재 작업 폴더와 플러그인 제작 규칙을 확인한 뒤, “공개 자료로 입증 가능하면서 데모가 강한 문제”를 골라 제출물까지 이어지는 형태로 설계하겠다. 플러그인 제작에는 `plugin-creator`, 최신 Codex 규격 확인에는 `openai-docs` 스킬을 사용할게.

## Codex (진행 메시지)

폴더는 아직 빈 상태라 방향을 깨끗하게 잡기 좋다. 지금은 아이디어를 넓게 벌리기보다, 심사에서 중요한 세 가지—공개 근거, 플러그인다운 반복 업무, 검증 가능한 전후 결과—가 동시에 잡히는 문제를 찾고 있어. 마이리얼트립 공식 자료와 고객이 실제로 마주치는 공개 정보 구조를 함께 대조하겠다.

## Codex

마이리얼트립에는 **“여행상품 등록 전 사전검수 플러그인”**이 가장 강한 주제라고 본다.

가칭: `MRT Product Preflight`

파트너가 CSV·JSON·Markdown 형태의 상품 초안을 넣으면 Codex가 다음을 검사한다.

- 필수 정보 누락
- 상품명 길이·금지 표현·가독성
- 카테고리·대표 도시·현지 시각 불일치
- 포함/불포함 사항과 일정의 모순
- 취소·환불 규정 충돌
- 번역 누락 및 원문과 번역문의 의미 차이
- 수정안, 오류 근거, 제출 가능 여부 생성

이 문제가 좋은 이유는 공개 근거가 매우 선명하기 때문이다.

- 마이리얼트립은 상품 온세일까지 정책 검수·카테고리 매칭·번역에 상당한 수작업이 필요하다고 밝혔다. [공식 기술 블로그](https://blog.myrealtrip.com/aicx-product-ops-automation/)
- 파트너가 등록한 2,500개 상품명이 길이와 표현 면에서 일관되지 않아 별도 최적화가 필요했다. [공식 AI 활용 사례](https://blog.myrealtrip.com/pepe-pm-title-optimization-2500/)
- 공식 등록 매뉴얼에는 상품 심사 중 정보를 수정할 수 없고, 상품명·카테고리·환불 규정 등에 구체적인 검수 기준이 명시돼 있다. [파트너 상품 등록 매뉴얼](https://mrtpartners.zendesk.com/hc/ko/article_attachments/31250849196045)

핵심 포지셔닝은 기존 AICX를 흉내 내는 것이 아니라, **파트너와 운영 담당자가 심사 요청 전에 오류를 발견하는 로컬 프리플라이트 도구**다. 이 구분이 있어야 “이미 회사가 해결한 문제”라는 반론을 피할 수 있다.

검증도 깔끔하다. 공개 매뉴얼에서 규칙을 추출해 정상·오류 샘플을 만들고, 플러그인 실행 전후의 탐지율·누락률·자동수정 결과를 테스트로 보여주면 된다. 단순 여행 일정 추천기보다 기업 문제, Codex 활용성, 정량 검증이 훨씬 잘 맞는다.

이 방향으로 확정하면 다음 단계에서 `submission.zip` 구조, 실제 플러그인, 테스트 데이터, README, 질문 5문항 답안까지 한 세트로 만들면 된다.

## Codex

전혀 벗어나지 않는다. 오히려 네가 말한 **“여행상품을 퍼즐처럼 조립한다”**는 관점이 기획의 핵심이 될 수 있다.

다만 그대로 “3박 4일 일정 추천 챗봇”이라고 정의하면 약하다. 마이리얼트립은 2023년에 이미 대화형 AI 여행플래너를 출시했고, 현재는 항공·숙박·투어 등을 조합하는 ‘마이팩’도 운영하고 있기 때문이다. [AI 여행플래너](https://www.mt.co.kr/future/2023/02/23/2023022314592163288), [마이팩 공식 회고](https://blog.myrealtrip.com/mypack-product-retrospective/)

차별점은 **추천이 아니라 조립과 검증**으로 잡으면 된다.

## 핵심 기획

가칭: `Trip Puzzle` 또는 `MyTrip Composer`

> 사용자가 선택한 항공편, 숙소, 티켓, 가이드 투어를 시간·동선·예산 조건에 맞게 조립하고, 실행 가능한 자유여행 일정으로 변환하는 Codex 플러그인

일반 여행 AI가 “오사카성을 가보세요”라고 추천한다면, 이 플러그인은 다음처럼 작동한다.

- 항공 도착: 1일 차 13:30 간사이공항
- 숙소: 난바
- 유니버설 스튜디오 티켓
- 교토 일일 가이드 투어: 3일 차 08:00
- 도톤보리 크루즈: 19:30
- 귀국 항공편: 4일 차 11:00

이를 받아서:

- 공항에서 숙소까지 이동시간 반영
- 체크인 전 짐 보관 시간 반영
- 상품 이용 가능 시간과 일정 충돌 검사
- 하루 이동량이 과도한 구간 경고
- 예약이 필요한 순서 정리
- 비어 있는 시간대만 추가 후보 제안
- 총예산과 예약 체크리스트 생성
- `itinerary.md`, `booking-checklist.md`, `calendar.ics` 생성

즉, **여행을 대신 결정하는 AI가 아니라 사용자가 선택한 여행을 실제로 실행 가능하게 만드는 컴파일러**에 가깝다.

## 공개적으로 입증할 수 있는 문제

마이리얼트립 공식 블로그가 네 문제의식을 거의 그대로 뒷받침한다.

마이리얼트립은 항공·숙박·투어·교통·티켓을 개별 제공하며 고객이 이를 직접 조합해 자유여행을 만들고 있지만, 그 과정에서 자유도를 얻는 대신 번거로움을 감수해야 한다고 설명한다. 또한 각 상품의 재고·가격·일정 구조가 달라 조합 가능성을 판단하는 일이 복잡하다고 밝히고 있다. [마이팩 공식 회고](https://blog.myrealtrip.com/mypack-product-retrospective/)

따라서 문제 정의는 이렇게 쓰면 좋다.

> 자유여행자는 자신이 원하는 항공·숙소·투어·티켓을 선택할 수 있지만, 서로 다른 예약 시간과 위치, 이동시간, 취소 조건을 직접 비교하며 하나의 실행 가능한 일정으로 조립해야 한다.

개인 경험은 문제를 발견한 계기로 쓰고, 공식 블로그를 객관적인 근거로 쓰면 된다.

## 기존 서비스와의 차이

| 기존 접근 | 우리 플러그인 |
|---|---|
| 여행지를 추천한다 | 사용자가 고른 상품을 조립한다 |
| 새로운 일정을 생성한다 | 기존 선택의 실행 가능성을 검증한다 |
| 오전·오후 수준의 일정 | 예약 시각·이동시간·버퍼까지 계산한다 |
| 결과를 대화로 보여준다 | 일정표·예약 체크리스트·캘린더 파일을 만든다 |
| 한 번 답하고 끝난다 | 상품 하나가 바뀌면 전체 일정을 다시 계산한다 |

## MVP 범위

첫 제출에서는 범위를 좁히는 것이 좋다.

- 목적지: 오사카
- 기간: 3박 4일
- 입력: 항공·숙소·선택한 투어/티켓 정보
- 핵심 기능:
  1. 일정 조립
  2. 시간 충돌 탐지
  3. 이동시간·휴식시간 검토
  4. 예약 체크리스트 생성
  5. 변경 요청에 따른 재배치
- 검증: 정상 여행안과 충돌 여행안으로 테스트

중요한 표현도 하나 있다. 과제에서는 이를 **GPT 플러그인**이 아니라 **Codex 플러그인**이라고 불러야 한다.

내 판단으로는 이 방향이 상품 사전검수 아이디어보다 네 경험과 문제의식이 더 잘 살아난다. 단, “AI 일정 추천”이 아니라 **“선택한 상품을 제약조건에 맞춰 조립·검증하는 자유여행 설계 도구”**로 밀어야 경쟁력이 생긴다.

## 사용자

근데 나는 이게 지금 이 해커톤에서 원하는 기능이 아닌것으로 보여서


## Codex

그 감각이 맞다. 문구상으로는 “기업 고객의 문제”라서 여행 플래너도 허용되지만, **해커톤 평가 관점에서는 위험한 주제**다.

이유는 세 가지다.

- 이미 마이리얼트립에 AI 여행플래너와 마이팩이 있어 신규성이 약하다.
- Codex의 강점인 파일 처리·반복 업무·검증·자동화를 충분히 활용하지 못한다.
- 결과물이 여행 추천 프롬프트나 일반 챗봇처럼 보일 가능성이 크다.

즉, 좋은 서비스 아이디어이지만 **좋은 Codex 플러그인 과제인지는 별개**다.

이 해커톤에 잘 맞는 문제는 보통 다음 조건을 갖는다.

1. 사용자가 반복적으로 수행하는 명확한 작업이 있다.
2. 입력 파일이나 데이터가 있다.
3. Codex가 분석·수정·검증한 결과물을 만든다.
4. 플러그인 사용 전후를 테스트로 비교할 수 있다.
5. 마이리얼트립이 아니면 설득력이 떨어지는 기업 특화 문제다.

이 기준으로 보면 이전의 **여행상품 등록 사전검수 플러그인**이 더 적합하다.

> 마이리얼트립 파트너가 상품을 심사 요청하기 전에 상품명, 카테고리, 일정, 포함·불포함 사항, 환불 정책을 자동으로 검사하고 수정안을 생성하는 Codex 플러그인

이 주제는 공식 매뉴얼을 규칙 데이터로 만들 수 있고, CSV·JSON 상품 파일을 실제로 검사하며, 오류 탐지율도 테스트할 수 있다. 결과적으로 “프롬프트 하나”가 아니라 실행 코드와 검증 체계를 가진 플러그인이 된다.

네가 말한 퍼즐 관점은 버릴 필요는 없다. 다음처럼 운영자 문제에 녹일 수도 있다.

> 여러 항공·숙소·투어 상품 데이터를 조합해 판매 가능한 자유여행 상품안을 만들고, 일정 충돌과 정책 오류를 검증하는 상품기획 플러그인

다만 이 경우 마이팩과 상당히 겹친다. 그래서 현재로서는 **상품 등록 사전검수**가 가장 안전하고, 일정 퍼즐은 후속 기능으로 넣는 구성이 좋다.

## 사용자

그러니까 사실상 내가 생각할땐 오히려 반복적인 업무를 간단히 하는것에 초점을 맞추는게 좋아보이거든 그럴려면 반복적인 업무가 무엇이 있을까를 고민해봐야할텐데 내가 지금 생각나는 건 가이드 한명 가이드 한명이 상품을 게시 했을때 상품이 겹치는 경우가 생각보다 많았거든 그런것을 그룹화해서 사용자가 보기 쉽게 데이터화하는 것을 만드는건 어때?


## Codex (진행 메시지)

이건 훨씬 Codex 플러그인답다. 핵심 반복 업무를 **“상품 카탈로그 중복·유사도 정리”**로 잡을 수 있고, 운영자가 수백·수천 개 상품 파일을 주기적으로 비교하는 흐름도 자연스럽다. 다만 ‘겹친다’를 정확히 정의해야 하니, 실제 공개 상품과 마이리얼트립이 밝힌 카탈로그 운영 문제를 기준으로 검증 가능성을 확인해보겠다.

## Codex

이 방향은 꽤 좋다. 특히 **“중복 상품을 제거한다”가 아니라 “유사 상품을 묶고 차이를 구조화한다”**로 정의하면 훨씬 설득력 있다.

같은 장소를 방문하더라도 가이드 언어, 차량, 소요시간, 픽업, 포함 비용, 취소 정책이 다르므로 단순 중복으로 판단하면 안 된다.

예를 들어 시드니 블루마운틴 상품은 핵심 방문지가 비슷하지만 다음 요소가 조금씩 다르다.

- 동물원 포함 여부
- 선셋·별보기 여부
- 소그룹 여부
- 출발 시간과 총 소요시간
- 식사·입장권 포함 여부

현재 공개 상품 목록에서도 비슷한 블루마운틴 상품이 한 화면에 다수 노출된다. [마이리얼트립 투어 목록](https://www.myrealtrip.com/experiences/)

여행자는 긴 제목을 하나씩 열어 차이를 찾아야 하고, 운영자는 신규 상품이 등록될 때마다 기존 상품과의 유사성을 반복해서 판단해야 한다.

## 반복 업무

운영자의 실제 반복 작업은 다음과 같이 정의할 수 있다.

1. 신규 또는 기존 상품 목록을 확인한다.
2. 도시·관광지·코스를 기준으로 유사 상품을 찾는다.
3. 상품 상세 내용을 하나씩 읽는다.
4. 가격·시간·이동수단·포함 사항을 비교한다.
5. 같은 상품군으로 묶을지 판단한다.
6. 고객에게 보여줄 차별점을 요약한다.
7. 기획전·검색·카테고리용 비교 데이터를 만든다.

마이리얼트립은 공급사마다 데이터 구조가 다르고 카테고리 매칭에도 수작업이 필요하다고 공개적으로 밝혔다. [상품 운영 자동화 사례](https://blog.myrealtrip.com/aicx-product-ops-automation/) 또한 파트너가 직접 작성한 2,500개 상품 제목의 가독성과 일관성이 문제였다고 설명한다. [상품명 최적화 사례](https://blog.myrealtrip.com/pepe-pm-title-optimization-2500/)

## 플러그인의 작동 방식

입력은 CSV·JSON 또는 상품 정보를 담은 Markdown 파일로 받는다.

플러그인이 상품마다 다음 속성을 추출한다.

- 국가·도시
- 핵심 관광지
- 전체 코스
- 상품 유형
- 소요시간
- 출발·종료 시각
- 이동수단
- 조인·단독 투어 여부
- 가이드 언어
- 픽업·샌딩
- 포함·불포함 사항
- 가격
- 취소 조건

그다음 상품을 세 종류로 분류한다.

- **높은 유사도:** 핵심 코스와 운영 방식이 거의 같음
- **부분 유사:** 일부 관광지는 같지만 구성에 차이가 있음
- **연관 상품:** 같은 지역이지만 서로 보완 가능한 상품

결과물은 다음처럼 만든다.

```text
교토·나라 일일 차량투어 그룹

공통점
- 오사카 출발
- 교토·나라 방문
- 약 9시간
- 차량 이동

주요 차이
- 상품 A: 단독 투어, 호텔 픽업, 일정 조정 가능
- 상품 B: 조인 투어, 정해진 코스, 낮은 가격
- 상품 C: 무료 스냅 촬영, 신형 차량 제공
```

그리고 파일로 출력한다.

- `clusters.json`: 상품 군집 데이터
- `comparison.md`: 고객용 비교표
- `review-queue.csv`: 운영자가 확인할 고유사도 상품
- `catalog-summary.md`: 상품군별 특징과 누락 정보

## 이 기획에서 중요한 원칙

상품 유사도가 높다고 자동 삭제하거나 통합하면 안 된다. 가이드별 서비스 차이가 실제 상품 가치이기 때문이다.

따라서 플러그인의 역할은:

> “중복을 판정하는 AI”가 아니라 “비슷해서 비교하기 어려운 상품을 사람이 판단할 수 있도록 구조화하는 AI”

로 잡는 것이 안전하다.

## 기획 정의

가칭: `MRT Catalog Trust`

## 사용자에게 주는 가치

- 제목만 다른 사실상 동일한 상품을 묶어서 확인
- 가격·코스·포함 사항·픽업 조건을 한눈에 비교
- 비슷해 보이지만 중요한 조건이 다른 상품 식별
- 잘못된 상품 선택과 중복 예약 가능성 감소
- 본인에게 맞는 가이드·투어를 더 빠르게 선택

## 마이리얼트립에 주는 가치

- 신규 상품과 기존 상품의 중복도를 자동 계산
- 동일 가이드의 불필요한 중복 등록 탐지
- 다른 계정에서 설명·코스를 복제한 정황 탐지
- 비정상적으로 낮거나 높은 가격 확인
- 상품명과 내용이 일치하지 않는 상품 탐지
- 과장 표현이나 필수 정보 누락 탐지
- 검토가 필요한 상품만 운영자에게 우선 제시

## 중요한 판정 구분

플러그인은 결과를 세 단계로 나누는 것이 좋다.

1. **고객 비교 대상**  
   유사하지만 선택 가능한 정상 상품

2. **중복 가능성**  
   코스·설명·가격 구조가 거의 동일하여 운영자 확인이 필요한 상품

3. **신뢰 검토 필요**  
   설명 복제, 비정상 가격, 과장 표현, 판매자 정보 불일치 등 이상 징후가 있는 상품

자동 삭제나 제재는 하지 않고 반드시 근거와 함께 사람에게 전달한다.

## 최종 문제 정의

> 마이리얼트립에는 서로 다른 가이드가 유사한 코스의 상품을 다양한 방식으로 등록한다. 이러한 다양성은 장점이지만, 여행자는 상품 간 핵심 차이를 파악하기 어렵고 운영자는 중복·복제·이상 상품을 반복적으로 검토해야 한다. 본 플러그인은 상품 정보를 구조화하고 유사상품을 그룹화하여 고객의 오인 선택을 줄이고, 운영자가 검토할 상품의 우선순위를 제시한다.

이 정도면 문제, 사용자 가치, 기업 가치, 반복 업무, AI 활용 이유가 하나의 흐름으로 연결된다. 이제 주제의 뼈대는 충분히 경쟁력이 있다.

## Codex

맞아. 다만 지금 문장은 **문제와 해결책이 섞여 있어서** 둘로 분리하면 더 명확하다.

문제 정의:

> 마이리얼트립에는 코스와 방문지가 유사한 투어가 개별 상품으로 나열되어 있어, 여행자가 상품 간 핵심 차이를 파악하기 어렵고 선택 착오가 발생할 수 있다.

해결책:

> 유사한 투어 상품을 하나의 카탈로그 또는 테마로 자동 그룹화하고, 가격·코스·포함 사항 등의 차이를 비교할 수 있도록 정리한다.

기업 측 문제까지 포함한 확장형은 다음과 같다.

> 여행자는 유사 상품의 차이를 파악하기 어렵고, 마이리얼트립 운영자는 중복·복제 가능성이 있는 상품을 반복적으로 검토해야 한다.

가장 추천하는 한 문장 버전은:

> **유사한 여행상품을 자동으로 그룹화하고 차이점을 구조화하여 여행자의 선택 착오와 운영자의 중복상품 검토 부담을 줄인다.**

‘중복 상품’보다는 **‘유사 상품’**이라는 표현이 좋다. 같은 코스라도 가이드와 서비스 조건이 다를 수 있기 때문이다.

## 사용자

이걸 만들었을 때 사용가치가 있을지를 생각해보면 결국 이거 고도화 작업을 통해 상품승인작업의 반복을 어느정도 줄일수도있지만 꼭 필요한 기능일까?

## Codex

솔직히 말하면, **유사 상품을 묶어 보여주는 기능만으로는 ‘꼭 필요한 기능’이라고 주장하기 어렵다.** 있으면 편리하지만 검색 필터나 추천 UI로도 해결할 수 있다는 반론이 가능하다.

더 강한 가치는 **상품 승인 업무의 1차 검수 자동화**에 있다.

## 방향 수정

핵심 문제:

> 신규 여행상품이 등록될 때 운영자는 상품 정보와 정책 준수 여부, 기존 상품과의 중복 가능성을 반복적으로 확인해야 한다.

플러그인의 역할:

> 신규 상품을 승인하기 전에 필수 정보, 정책 위반, 유사·복제 가능성, 정보 간 모순을 검사하여 심사 보고서를 생성한다.

유사상품 그룹화는 이 플러그인의 핵심 검사 항목 중 하나로 넣는다.

### 입력

- `new-product.json`: 신규 등록 상품
- `existing-catalog.csv`: 기존 상품 목록
- `review-policy.md`: 상품 등록·심사 기준

### 자동 검사

- 필수 항목 누락
- 상품명 길이와 금지 표현
- 상품 설명과 카테고리 불일치
- 포함·불포함 사항의 모순
- 취소·환불 정책 누락
- 기존 상품과의 유사도
- 다른 판매자의 설명 복제 가능성
- 동일 상품의 반복 등록 가능성
- 가격 이상치

### 출력

```text

필수 수정

- 판매자는 서로 다름

권장 조치
- 설명 작성 경위 확인
- 기존 상품과의 차별점 추가 요청
- 자동 승인하지 않고 수동 검토
```

## 사용가치 판단

| 기능 | 필요성 |
|---|---:|
| 고객용 유사상품 그룹화 | 있으면 편리함 |

따라서 최종 기획은 이렇게 잡는 것이 좋다.

> **마이리얼트립 신규 여행상품의 정책 준수와 기존 상품 중복 가능성을 자동으로 검사하여 상품 승인 담당자의 반복 검수 업무를 줄이는 Codex 플러그인**

이렇게 하면 “왜 꼭 필요한가?”에 대해 **승인 업무 시간 단축, 심사 기준 일관성, 위험 상품 조기 발견**이라는 명확한 답을 줄 수 있다. 자동 승인 도구가 아니라 담당자의 판단을 빠르게 만드는 심사 보조 도구라는 점도 중요하다.

## 사용자

그러니까 비판을 해보자면 기능을 위한 기능 즉 결국은 사람이 해야하는 비효율성을 촉발할수도있어보여 그리고 오히려 진짜 재대로 된곳도 커트하는 문제가 발생할수도 있어보이거든 반대로 재대로 못거르면 효여성이 떨어져 따라서 명확한 앵커가 필요해보이고 진짜 이게 효율적이라면 스몰데이터를 활용해서 겹치는 상품이 많은지 확인해봐야겠네

## Codex

맞아. 지금 단계에서 바로 만들면 **“AI를 쓰기 위해 만든 기능”**이 될 위험이 크다.

핵심 위험은 정확히 네가 짚은 두 가지다.

- 오탐이 많으면 정상 상품까지 검토 대상으로 보내 사람의 일을 늘린다.
- 오탐을 줄이려고 기준을 느슨하게 하면 기존 수작업과 차이가 없다.

따라서 ‘사기 탐지’나 ‘승인 자동화’부터 주장하면 안 된다. 먼저 다음 가설 하나만 검증해야 한다.

> 특정 여행지·투어 유형에서 핵심 코스가 겹치는 상품이 실제로 충분히 많고, 상품 간 차이 정보가 비구조적으로 흩어져 있어 비교 비용이 발생하는가?

## 스몰데이터 검증안

표본은 40~60개 정도면 초기 판단이 가능하다.

- 시드니 블루마운틴 투어 20개
- 오사카 출발 교토·나라 투어 20개
- 비교군으로 상품 다양성이 높은 다른 카테고리 20개

각 상품에서 다음 항목만 수집한다.

- 상품명
- 방문 장소와 코스
- 투어 시간
- 조인·단독 여부
- 이동수단
- 픽업·샌딩
- 포함·불포함 사항
- 가격
- 취소 조건
- 판매자

## ‘겹친다’의 기준

텍스트가 비슷하다는 이유만으로 중복 처리하면 안 된다. 다음처럼 계층화해야 한다.

- **동일 경험군:** 핵심 방문지와 투어 방식이 대부분 같음
- **유사 경험군:** 방문지는 겹치지만 시간·이동수단·서비스가 다름
- **별도 상품:** 같은 지역이지만 목적과 경험이 다름
- **판단 불가:** 정보가 부족함

초기 검증에서는 AI가 아니라 사람이 먼저 라벨링해야 한다. 이것이 정답 데이터가 된다.

## 계속할지 결정하는 앵커

다음 기준을 미리 정해두면 결과를 억지로 해석하지 않을 수 있다.

- 전체 상품 중 30% 이상이 3개 이상의 유사상품군에 속하는가?
- 동일 상품군에서 비교해야 할 핵심 조건이 일관되게 표기되지 않는가?
- 사람이 부여한 그룹과 플러그인의 그룹이 90% 이상 일치하는가?
- 플러그인 결과가 수동 비교 시간을 절반 이상 줄이는가?
- 플러그인이 새로운 검토 업무를 추가하지 않는가?

기준을 충족하지 못하면 이 아이디어는 버리는 게 맞다.

그리고 현재 단계에서는 **상품 승인이나 제재 기능은 제외**하는 편이 좋다. 먼저 “유사상품이 실제로 많은가”와 “그룹화가 정말 비교를 쉽게 만드는가”만 검증해야 한다. 그 결과가 유의미할 때 운영자용 검토 기능을 확장하는 순서가 안전하다.

그럼 일단 지금내 단계에서는 스코프를 정하기 보다는 내 가설이 맞는지를 먼저 검증하고 시작하자

## Codex (진행 메시지)

좋아. 지금은 아이디어를 지키는 단계가 아니라 **틀릴 수 있게 검증하는 단계**로 가자. 먼저 가설과 탈락 기준을 고정한 다음, 공개된 마이리얼트립 상품을 소규모로 표본화해 실제 유사상품 비율과 비교정보의 비구조성을 측정하겠다. 결과가 약하면 미련 없이 방향을 바꾸는 전제로 진행할게.

## Codex (진행 메시지)

첫 표본에서 중요한 반증이 바로 나왔다. 일부 상품 페이지에 이미 **“비슷한 상품과 비교해보세요”** 영역이 존재한다. 즉 “유사상품을 묶는 기능이 없다”는 가설은 그대로는 성립하지 않을 가능성이 크다. 이제 검증 초점을 ‘기능 존재 여부’가 아니라 **기존 비교가 어떤 기준으로 묶이고, 핵심 차이를 실제로 구조화하는지**로 바꾸겠다. 이런 반증이 일찍 나온 건 오히려 좋은 신호다.

## Codex (진행 메시지)

현재 페이지 4개를 파일럿으로 보면 기존 추천은 ‘비슷함’의 앵커가 일관되지 않다. 같은 관광지·다른 상품형태, 같은 도시·전혀 다른 체험, 같은 활동·다른 국가가 섞인다. 그래서 검증 문서를 만들 때 원가설과 수정가설을 분리하겠다:  
1) 유사상품이 충분히 많은가, 2) 기존 비교가 구매 결정에 필요한 차이를 제대로 구조화하는가.  
또한 사이트가 자동 스크래핑을 금지하므로, 대량 수집 대신 공개 검색 결과를 이용한 제한적 수동 표본으로 진행한다.

## Codex (진행 메시지)

가설의 첫 절반은 확인됐다. 인기 경험군에는 실제로 비교 가능한 상품이 여러 개 존재한다. 하지만 더 중요한 결론은 **단순 그룹화만으로는 부족하다**는 것이다. 같은 교토·나라 상품도 버스투어와 프라이빗 차량투어를 한 그룹으로 뭉치면 오히려 혼란이 생긴다. 마지막으로 대표 상품 5개의 핵심 필드가 공통 기준으로 실제 추출되는지 확인해, 비교표 자체가 성립하는지 보겠다.

## 사용자

자 그럼 정리하면 지금 이 플러그인이 필요한가에 대한 명제에는 o가되는지를호ㅓㄱ인해보자

## Codex

결론부터 말하면 현재 판정은 **O가 아니라 △**다.

| 검증 항목 | 현재 판정 |
|---|---|
| 유사한 상품이 충분히 존재하는가 | O |
| 기존 비교 후보가 항상 적절한가 | X |
| 사용자가 실제로 선택 착오를 겪는가 | 미검증 |
| 구조화 비교가 선택을 개선하는가 | 미검증 |
| 이를 Codex 플러그인으로 해결해야 하는가 | △ |

근거는 있다.

- 오사카 표본 12개 중 8개가 ‘교토·나라 당일투어’ 상위 경험군에 속했다.
- 블루마운틴에서도 직접 비교 가능한 상품군이 충분히 깊었다.
- 기존 `비슷한 상품과 비교해보세요` 추천 18개를 살펴본 파일럿에서는 6개가 여행지나 구매 목적상 직접 비교하기 어려웠다.
- 반면 마이리얼트립에는 이미 유사상품 비교 영역이 존재한다. 따라서 단순 그룹화 플러그인은 중복 기능이다.

즉, 현재 확인된 명제는 다음이다.

> 인기 여행지에는 유사상품이 많고 기존 비교 후보에 일부 불일치가 있다. — O

하지만 우리가 증명해야 할 명제는 더 강하다.

> 여행자가 유사상품의 차이를 파악하는 데 실제 어려움을 겪으며, 구조화 비교표가 기존 방식보다 선택 시간과 오류를 유의미하게 줄인다. — 아직 미검증

따라서 지금 플러그인을 만들면 안 된다. 다음 실험에서 O가 나와야 한다.

1. 교토·나라 투어 5개를 고른다.
2. 사용자 3~5명에게 기존 상품 페이지만 보고 조건에 맞는 상품을 선택하게 한다.
3. 선택 시간, 열어본 상세 페이지 수, 핵심 조건을 잘못 이해한 횟수를 기록한다.
4. 동일 상품을 구조화 비교표로 제공하고 다시 측정한다.
5. 비교표가 시간을 줄이면서 선택 정확도를 높이는지 확인한다.

여기서 개선이 없다면 기각이다. 개선이 명확하면 그때 비로소 “필요한 플러그인”에 O를 줄 수 있다.

## 사용자

근데 비슷한 항목에 비슷한조회수라면 이게 옳다는거아닌가?

## Codex

일부는 맞지만, 그것만으로 “옳다”고 확정할 수는 없어.

**비슷한 상품이 비슷한 조회수를 가진다**는 것은 다음 정도를 의미한다.

> 두 상품이 비슷한 수요를 받거나 같은 고객군에 노출되고 있을 가능성이 있다.

하지만 이것이 곧 “사용자가 비교하기 어려워한다”는 증거는 아니다. 오히려 기존 추천이 이미 잘 작동해서 조회수가 비슷할 수도 있다.

예를 들면:

- 뉴포트비치 고래투어와 마데이라 고래투어가 조회수는 비슷해도 여행지가 달라 대체상품이 아니다.
- 교토·나라 버스투어 두 개가 조회수도 비슷하고 코스도 비슷하다면 직접 비교군일 가능성이 높다.
- 하지만 두 상품의 전환율과 취소율까지 비슷하다면 현재 선택 과정에 큰 문제가 없을 수도 있다.

필요성을 증명하려면 조회수 다음 행동이 중요하다.

- 여러 유사 상품 페이지를 반복해서 이동하는가
- 상세 페이지를 많이 열고도 구매하지 않는가
- 상품 조건을 잘못 이해한 문의가 발생하는가
- 구매 후 취소하거나 다른 상품으로 바꾸는가
- 비교표를 제공했을 때 선택 시간이 줄고 정확도가 높아지는가

그리고 공개 페이지에서 보이는 숫자는 대부분 **조회수보다 후기 수**에 가깝다. 후기 수는 판매 기간, 광고, 가격, 노출 순위 영향을 받으므로 조회수처럼 해석하면 안 된다.

정리하면:

> 유사한 상품의 조회수가 비슷하다 → 비교 대상으로 묶을 근거는 될 수 있음  
> 유사한 상품의 조회수가 비슷하다 → 비교 플러그인이 필요하다는 증거는 아님

우리에게 필요한 진짜 앵커는 **유사상품을 여러 번 조회하지만 핵심 차이를 파악하지 못해 선택 비용이 커진다**는 것이다. 조회수와 함께 페이지 이동·전환 또는 사용자 실험 결과가 있어야 O 판정을 내릴 수 있다.

## Codex

여기서 중요한 오해가 하나 있어. 이번 과제는 **GPT 플러그인이 아니라 Codex 플러그인**이다.

사주·그림 그리기처럼 일반 사용자가 ChatGPT에서 사용하는 GPT와는 다른 체계다. Codex 플러그인은 반복 가능한 스킬, 스크립트, MCP 연동 등을 묶어 **Codex 사용자에게 설치하는 작업 도구**다. 공식 문서도 플러그인을 “다른 개발자가 설치할 수 있는 재사용 가능한 스킬과 앱의 배포 단위”로 설명한다. [Codex 스킬 문서](https://developers.openai.com/codex/skills), [플러그인 구조](https://developers.openai.com/codex/plugins/build#plugin-structure)

따라서 마이리얼트립 사원이 사용하는 플러그인은 전혀 이상하지 않다. 오히려 일반 여행자가 여행 중 Codex를 실행해 플러그인을 쓰는 시나리오가 더 부자연스러울 수 있다.

현실적인 사용자 적합도는 다음과 같다.

| 사용자 | Codex 플러그인 적합도 |
|---|---:|
| 마이리얼트립 개발자 | 매우 높음 |
| 데이터·상품·콘텐츠 운영자 | 높음 |
| 여행상품 파트너·가이드 | 업무가 파일 기반이면 가능 |
| 일반 여행자 | 상대적으로 낮음 |
| ChatGPT에서 여행 추천을 원하는 사용자 | GPT나 ChatGPT 앱이 더 적합 |

과제에서 “기업의 고객 문제도 가능하다”고 한 것은 고객이 직접 플러그인을 써야 한다는 뜻이 아니다. 직원이 Codex 플러그인으로 고객 문제를 분석하거나 서비스 품질을 개선해도 된다.

따라서 우리가 찾을 방향은 다시 이렇게 정리된다.

> **마이리얼트립 사원·파트너가 Codex에서 반복적으로 수행할 만한 업무이면서, 그 결과가 고객 경험이나 기업 효율을 개선하는가?**

내가 앞에서 일반 소비자용 GPT와 Codex 플러그인의 관점을 섞어 말한 부분은 바로잡아야 한다. 지금까지 나온 것 중에서는 여행자용 더치페이 도구보다 상품 검수·예약 QA·콘텐츠 운영처럼 **파일과 규칙을 다루는 내부 업무**가 과제 형식에 더 자연스럽다.

> 마이리얼트립 기획자가 고객 문제 아이디어를 입력하면 공개 자료와 소규모 데이터를 바탕으로 문제의 실재성, 기존 기능, 가설, 반증 조건을 정리하고 개발 여부를 판단하도록 돕는 Codex 플러그인

입력:

```text
친구와 투어를 예약할 때 대표자 한 명만 결제해야 해서 정산이 불편했다.
```

```text
문제 근거

- 공식 자료: 부분 확인
- 다수 사용자 근거: 부족

기존 해결책
- 원결제수단 환불 지원
- 분할결제 지원 여부는 확인되지 않음

- 다인 예약에서 대표 결제자의 정산 부담이 구매 경험을 저해한다.

반증 조건
- 대부분 균등 분할로 간단히 해결된다.
- 기존 송금 서비스로 충분하다.
- 다인 예약 비중이 낮다.

현재 판정
- 문제 존재: △
- Codex 플러그인 적합성: X
- 마이리얼트립 제품 기능 후보: △
```

결과 파일은 다음처럼 만들 수 있다.

- `evidence-ledger.md`: 사실·추론·미확인 사항
- `existing-feature-map.md`: 기존 기능과 중복 여부
- `hypothesis.md`: 검증·반증 가능한 가설
- `small-data-plan.md`: 최소 검증 방법
- `decision.md`: 진행·수정·기각 판정
- `prd-draft.md`: 검증 통과 시에만 생성되는 기획 초안

이 플러그인의 핵심 가치는 **기획서를 빨리 작성하는 것**이 아니다.

> 근거가 약한 아이디어가 기능 개발로 넘어가는 것을 막고, 기획자가 문제를 먼저 검증하게 만드는 것

실제로 우리는 유사상품 아이디어를 조사하면서 기존의 `비슷한 상품과 비교해보세요` 기능을 뒤늦게 발견했다. 이런 중복과 성급한 기능화를 초기에 잡아주는 것이 명확한 사용 가치다.

다만 이것도 바로 확정하면 안 된다. 다음 검증 대상은 “마이리얼트립 또는 여행 플랫폼 기획자가 아이디어 검증 과정에서 공개 근거 수집과 기존 기능 확인을 반복적으로 수행하는가?”가 된다. 방향 자체는 지금까지 나온 후보 중 네 역량과 Codex 형식이 가장 자연스럽게 만난다.

## 사용자

스코프를 줄여서 사용자는 마이리얼트립 기능 기획자로 줄이면 쉽겠네

## Codex

맞아. 사용자 범위를 이렇게 고정하면 된다.

> **마이리얼트립에서 신규 기능을 제안하거나 검토하는 프로덕트 기획자(PM)**

사용 상황도 한정한다.

> 고객 불편이나 기능 아이디어를 발견했지만, PRD 작성과 개발 요청 전에 실제 문제인지 검증해야 하는 상황

플러그인의 단 하나의 역할:

> 기능 아이디어 하나를 입력받아 기존 기능, 공개 근거, 반증 자료와 최소 검증 방법을 정리한다.

범위에서 제외할 것:

- 완성된 PRD 자동 작성
- 화면 디자인
- 개발 구현
- 전체 로드맵 우선순위 결정
- 근거 없는 AI 기능 추천
- 비공개 고객정보 분석

핵심 출력도 세 개면 충분하다.

1. `evidence.md`: 근거·반증·미확인 사항  
2. `validation-plan.md`: 스몰데이터 검증 방법  
3. `decision.md`: 진행·수정·기각 판정  

한 문장으로 정리하면:

> **마이리얼트립 기능 기획자가 개발 전에 아이디어의 필요성과 기존 기능 중복 여부를 공개 자료로 검증하도록 돕는 Codex 플러그인**

이제 사용자와 사용 시점, 작업 범위가 상당히 명확해졌다. 다음은 “기획자가 실제로 이런 검증 과정에서 어떤 반복 업무를 겪는가”만 입증하면 된다.

## 사용자

그럼 지금 우리가 한 일련의 과정이 될수있겠네 


## Codex

맞아. **지금 우리가 대화에서 수행한 일련의 사고 과정 자체가 플러그인의 핵심 워크플로우**가 될 수 있다.

우리가 실제로 한 일은 다음과 같다.

1. 개인 경험에서 불편 발견
2. 문제와 해결책 분리
3. Codex 플러그인에 적합한지 판단
4. 기존 기능 조사
5. 공개 근거 수집
6. 반대 논리와 오탐 위험 검토
7. 소규모 데이터로 가설 확인
8. `진행 / 수정 / 홀딩 / 기각` 결정
9. 검증을 통과한 경우에만 기획서 작성

이를 플러그인의 고정 절차로 만들면 된다.

```text
아이디어 입력
→ 문제/해결책 분리
→ 기존 기능 검색
→ 공개 근거와 반증 수집
→ 스몰데이터 검증안 생성
→ 필요성·기업특화성·Codex 적합성 평가
→ GO / REVISE / HOLD / DROP 판정
```

가장 좋은 데모도 이미 확보했다.

### 데모 입력

> 유사한 마이리얼트립 상품을 하나의 테마로 묶어 여행자의 선택 착오를 줄이고 싶다.

### 플러그인 결과

> HOLD  
> 마이리얼트립에는 이미 ‘비슷한 상품과 비교해보세요’ 기능이 존재한다. 단순 그룹화는 중복 기능이다. 기존 비교 후보의 관련성과 구조화 수준을 먼저 검증해야 한다.

이 결과의 가치는 “멋진 기능을 제안했다”가 아니다.

> **불필요한 기능 개발을 시작하기 전에 멈추게 했다.**

플러그인의 정체성은 다음처럼 잡을 수 있다.

> 마이리얼트립 기능 기획자가 고객 불편을 바로 기능으로 연결하지 않고, 공개 근거와 반증을 통해 개발 필요성을 검증하도록 돕는 프리-PRD 의사결정 플러그인

이 방향은 대화 로그와 플러그인 기능의 정합성도 매우 높다. 지금 로그 자체가 우리가 왜 이 도구가 필요하다고 생각했는지를 보여주는 개발 과정이 된다. 다만 제출용 로그는 과제 지침대로 편집하지 않고 원본 전체를 그대로 보존해야 한다.

## 사용자

그러면 이런 일련의 기획과정에서 에이전트가 할수있도록 범위를 정하고 어떻게 반례를 제시할지등을 정해야겠다. 이거를 CAPM에서 했듯이 Project Charter로 제작하는 걸 명확히 할수있도록 단 여기서 중요포인트는 기획자 개발자 그리고 디자이너, CEO모두 특화식으로 만들어야 겠네 


## Codex

맞아. 다만 사용자 범위는 계속 **마이리얼트립 기능 기획자**로 유지하고, 개발자·디자이너·CEO는 플러그인의 사용자가 아니라 **검토 관점**으로 넣는 것이 좋다. 그래야 스코프가 다시 퍼지지 않는다.

핵심 산출물은 일반 Project Charter보다 조금 앞단에 있는 가칭 **Feature Validation Charter**로 잡을 수 있다.

> 기능 개발을 승인하기 전에 문제의 실재성, 반례, 역할별 관점, 검증 계획을 명시하는 기획 헌장

## 에이전트의 작업 범위

```text
아이디어 입력
→ 문제와 해결책 분리
→ 공개 근거 조사
→ 기존 기능·대안 조사
→ 핵심 가설 작성
→ 반례와 실패 조건 생성
→ 역할별 검토
→ 최소 검증 계획 수립
→ Feature Validation Charter 생성
→ GO / REVISE / HOLD / DROP
```

## 역할별 특화 관점

| 관점 | 에이전트가 검토할 질문 |
|---|---|
| 기획자 | 누구의 어떤 문제가 얼마나 자주 발생하는가? 더 작은 해결책은 없는가? |
| 디자이너 | 사용자 여정 어디에서 문제가 발생하는가? 새 기능이 오히려 복잡성을 높이지 않는가? |
| 개발자 | 필요한 데이터와 연동은 무엇인가? 예외 상황·보안·기술부채는 무엇인가? |
| CEO·경영진 | 회사 전략과 연결되는가? 기회비용보다 가치가 큰가? 언제 중단할 것인가? |

사용자가 “더치페이가 불편하다”를 입력하면 다음처럼 서로 다른 반례가 나온다.

- 기획자: 기존 송금 서비스로 충분하지 않은가?
- 디자이너: 분할결제 단계가 예약 전환율을 낮추지 않는가?
- 개발자: 한 명이 결제를 완료하지 않으면 재고를 언제까지 점유할 것인가?
- 경영진: 결제 복잡도와 운영비 증가보다 실제 전환 개선이 큰가?

## 반례 생성 규칙

반례는 단순한 악마의 변호사가 아니라 다음 형식으로 만들어야 한다.

```text
주장:
다인 예약에서 대표 결제가 사용자에게 큰 불편을 준다.

가장 강한 반례:
대부분의 사용자는 기존 송금 기능으로 즉시 정산하므로
마이리얼트립 내부 기능이 필요하지 않을 수 있다.

현재 근거:
개인 경험만 존재하며 다수 사용자 자료는 확인되지 않음.

반례가 사실일 때 영향:
기능 개발 필요성 기각

검증 방법:
다인 예약자 인터뷰 또는 정산 관련 문의·취소 데이터 확인

상태:
미검증
```

모든 핵심 주장에 반드시 다음 다섯 항목이 붙어야 한다.

- 근거
- 가장 강한 반례
- 미확인 정보
- 검증 방법
- 기각 조건

## Feature Validation Charter 구조

```markdown
# Feature Validation Charter

## 1. 제안 개요
## 2. 대상 사용자와 사용 상황
## 3. 해결하려는 문제
## 4. 공개 근거와 출처
## 5. 기존 기능과 대체 해결책
## 6. 핵심 가설
## 7. 범위와 비범위
## 8. 역할별 검토
### 기획자 관점
### 디자이너 관점
### 개발자 관점
### 경영진 관점
## 9. 반례 및 리스크 등록부
## 10. 최소 검증 계획
## 11. 성공 지표와 중단 조건
## 12. 의사결정
GO / REVISE / HOLD / DROP
```

중요한 원칙은 Charter가 아이디어를 그럴듯하게 포장하는 문서가 되어서는 안 된다는 것이다.

> **개발을 승인하기 위한 문서인 동시에, 근거가 부족하면 개발을 중단시키는 문서**

이렇게 정의하면 네 CAPM 경험과 기능 기획 역량이 플러그인의 고유한 방법론으로 들어간다. 여러 역할을 위한 기능을 각각 만드는 대신, 하나의 기획안을 네 가지 관점으로 압박 검토하는 구조라서 스코프도 유지된다.

## 사용자

근데 오히려 이걸 다시 돌아가서 체크리스트로 만드는게 낫지 않을까?


## Codex

맞아. **Project Charter부터 만들면 AI가 근거가 약한 아이디어도 그럴듯한 문서로 포장할 위험**이 있다. 이 단계에서는 체크리스트가 더 적합하다.

플러그인의 역할을 다음처럼 줄이면 된다.

> 마이리얼트립 기능 기획자가 PRD 작성 전에 아이디어의 필요성·중복성·실현 가능성을 역할별 체크리스트로 검증한다.

## Pre-PRD 체크리스트

### 1. 기획자 관점

- 대상 사용자가 한 문장으로 정의됐는가?
- 문제와 해결책이 분리됐는가?
- 개인 경험 외에 공개 근거가 있는가?
- 문제의 발생 빈도나 영향도를 확인했는가?
- 기존 기능과 중복되지 않는가?
- 기능 추가 없이 해결할 방법은 없는가?

### 2. 디자이너 관점

- 문제가 발생하는 사용자 여정이 특정됐는가?
- 사용자가 현재 사용하는 우회 방법이 확인됐는가?
- 새 기능이 단계를 추가해 더 복잡하게 만들지 않는가?
- 예외 사용자나 접근성 문제가 고려됐는가?

### 3. 개발자 관점

- 필요한 데이터가 실제로 존재하는가?
- 외부 API나 내부 시스템 의존성이 확인됐는가?
- 오탐·오류가 발생했을 때 피해가 정의됐는가?
- 최소 기능으로 검증할 수 있는가?
- 테스트 가능한 성공 조건이 있는가?

### 4. 경영진 관점

- 마이리얼트립의 고객 또는 사업 문제와 연결되는가?
- 예상 가치가 개발·운영 비용보다 큰가?
- 더 중요한 문제를 미루게 만드는 기회비용은 없는가?
- 실패를 판단하고 중단할 조건이 있는가?

## 항목별 기록 형식

모든 항목은 단순 체크가 아니라 근거까지 남긴다.

```text
질문: 기존 기능과 중복되지 않는가?
상태: FAIL
근거: 상품 페이지에 ‘비슷한 상품과 비교해보세요’ 기능이 존재함
반례: 기존 기능은 구조화 비교가 아니라 추천 카드에 불과할 수 있음
다음 행동: 기존 비교 기능의 정확성과 사용자 효용 검증
```

상태는 네 개만 사용한다.

- `PASS`: 근거 확인
- `FAIL`: 반대 근거 확인
- `UNKNOWN`: 확인할 자료 부족
- `N/A`: 해당 없음

## 판정 규칙

- 핵심 항목에 `FAIL` → `DROP` 또는 `REVISE`
- 핵심 항목이 `UNKNOWN` → `HOLD`
- 반례 검증이 남음 → `HOLD`
- 핵심 항목이 모두 `PASS` → `GO`
- `GO` 이후에만 Project Charter 또는 PRD 생성

최종 산출물도 하나면 충분하다.

```text
feature-readiness-checklist.md
```

결론적으로 구조는 다음이 가장 좋다.

> **아이디어 → 역할별 체크리스트 → 근거·반례 → GO/HOLD/REVISE/DROP → 통과 시 Charter**

이 방식이 문서 제작 플러그인보다 훨씬 날카롭고, 우리가 실제로 겪은 시행착오도 그대로 반영한다.

## 사용자

그러니까 이런식으로 할수있도록 하는게 좋아 보여 그러니까 이게 하나의 시스템이 되는거지


## Codex

맞아. 단순 체크리스트 생성기가 아니라 **기능 아이디어를 검증하는 의사결정 시스템**이 되는 거야.

시스템 흐름은 이렇게 고정할 수 있다.

```text
아이디어 접수
→ 문제·해결책 분리
→ 근거 수집
→ 기존 기능 확인
→ 역할별 체크
→ 반례 생성
→ 미확인 가설 추출
→ 최소 검증 계획
→ GO / REVISE / HOLD / DROP
→ GO일 때만 Charter 작성
```

핵심 구성요소는 다섯 개다.

1. **Idea Intake**  
   사용자, 상황, 불편, 제안 기능을 구조화한다.

2. **Evidence Ledger**  
   모든 내용을 `사실 / 추론 / 개인 경험 / 미확인`으로 구분한다.

3. **Role Review**  
   기획자·디자이너·개발자·경영진 관점의 체크리스트를 적용한다.

4. **Counterexample Engine**  
   주장마다 가장 강한 반례, 실패 가능성, 더 작은 대안을 제시한다.

5. **Decision Gate**  
   근거와 반례에 따라 다음 단계 진입 여부를 결정한다.

```text
GO      충분한 근거가 있어 Charter 작성
REVISE  문제 또는 해결책을 수정
HOLD    핵심 정보가 부족해 검증 필요
DROP    중복 기능이거나 가치가 낮음
```

체크리스트는 시스템의 화면이고, 실제 핵심은 **근거 없이 다음 단계로 넘어가지 못하게 하는 게이트 구조**다.

Codex 플러그인도 이에 맞춰 단순하게 구성할 수 있다.

- `SKILL.md`: 전체 검증 절차

기획자·디자이너·개발자·경영진은 각각 다른 사용자가 아니라, **한 명의 PE가 반드시 검토해야 할 네 가지 관점**이 된다.

- 기존 기능 자동 조사
- 가장 강한 반례 제시
- 핵심 항목이 `UNKNOWN`이면 진행 차단
- `GO`일 때만 Charter 생성
- 마이리얼트립의 고객·파트너·예약·결제·상품 운영 맥락 반영

현재 판정은 **GO**다. 다만 바로 코드부터 만들기보다는 다음 단계에서 체크리스트 항목과 판정 규칙을 먼저 명세해야 한다. 이 규칙이 플러그인의 진짜 제품이기 때문이다.

## Codex

맞아. 정확히는:

> **마이리얼트립 Product Engineer의 기능 아이디어 초기 단계를 관리하는 Codex 플러그인**

아이디어를 대신 만들어주는 도구가 아니라, 막연한 아이디어가 개발 단계로 성급하게 넘어가지 않도록 관리한다.

```text
아이디어 등록
→ 문제와 해결책 분리
→ 근거·반례 확인
→ 역할별 체크리스트
→ 검증 과제 관리
→ GO / REVISE / HOLD / DROP
→ GO만 Charter·PRD 단계로 전달
```

핵심 출력은 다음 정도다.

- 현재 상태
- 확인된 근거
- 가장 강한 반례
- 미확인 가설
- 다음 검증 행동
- 개발 단계 진입 가능 여부

범위도 명확하다.

- 아이디어 발상 도구: 아님
- PRD 자동 작성기: 아님
- 프로젝트 관리 도구: 아님
- 개발 구현 도구: 아님
- **Pre-PRD 아이디어 검증·게이트 시스템: 맞음**

가칭은 `Idea Gate`, `Pre-PRD Gate`, `MRT Idea Check` 정도가 어울린다. 핵심 문장은 **“좋은 아이디어를 만드는 것보다, 검증되지 않은 아이디어가 개발되는 것을 막는다”**로 잡을 수 있다.

| 결과가 무거움 | 잘못된 아이디어의 구현 비용을 고려하면 정밀 검토 자체는 가치가 있음 |
| 엄격한 추론 제거 | 돈과 실행이 걸린 기획이라면 오히려 필요한 안전장치임 |

평가 우려가 아이디어 표현을 감소시킬 수 있다는 연구가 있고, 컴퓨터 기반 아이디어 평가를 그 완화 수단으로 제안한 연구도 있어. 다만 이것이 우리 플러그인이 실제로 자신감과 소통을 향상한다는 증거까지는 아니야. [평가 우려와 AI 아이디어 평가 연구](https://doi.org/10.1080/0144929X.2021.2023638), [평가 우려와 공동 아이디어 생성 실험](https://pmc.ncbi.nlm.nih.gov/articles/PMC6620827/)

마이리얼트립도 공식적으로 사람이 “무엇을·왜 만들지” 결정하고 결과를 검증하며, 혼자 해결하기보다 팀과 방향을 맞추는 것이 중요하다고 설명해. 이 부분은 플러그인과 직접 연결할 수 있어. [마이리얼트립 Product Engineer 사례](https://blog.myrealtrip.com/pepe-ai-codes-pe-thinks/)

### 새롭게 드러난 핵심 위험

가장 큰 문제는 이름과 판정 언어야.

`Idea Gate`, `GO`, `DROP`은 평가받는 두려움을 낮추기보다 오히려 강화할 수 있어. 사용자가 AI의 `DROP`을 권위 있는 평가로 받아들이면 자신감을 얻는 대신 좋은 아이디어까지 스스로 묻어버릴 수 있거든.

또한 AI가 주는 확신이 반드시 정확성을 뜻하지는 않아. 생성형 AI와 창의적 자신감에 관한 연구도 결과가 혼재하며, AI가 보조자로 작동할 때는 도움이 될 수 있지만 주 평가자로 작동하면 자신감을 떨어뜨리거나 자기 판단을 흐릴 수 있다고 정리해. [생성형 AI와 창의적 자기효능감 연구](https://www.sciencedirect.com/science/article/pii/S1871187126001094)

따라서 설계 원칙은 이렇게 바꾸는 게 좋아 보여.

- AI는 아이디어의 합격·불합격을 판정하지 않는다.
- 사용자가 팀과 대화할 준비가 되었는지를 보여준다.
- `HOLD`는 종료가 아니라 근거를 보충해 다시 실행하는 상태다.
- 내부 자료·사용자 경험·테스트 결과를 이어서 추가할 수 있어야 한다.
- 최종 출력은 “좋은 아이디어인가?”보다 “무엇을 사실로 말할 수 있고, 무엇을 질문으로 가져가야 하는가?”에 집중한다.

결국 가장 적합한 핵심 질문은 이것 같아.

이렇게 정의하면 네가 말한 자신감, 소통, 검증 기회의 논리가 하나로 연결돼.

### 미리 정할 기준

- 다음 단계로 이어갈 조건: 현재 탐색에서 기준 콘텐츠 누락 또는 외부 이동이 반복적으로 관찰됨
- `DROP` 조건: 현재 탐색으로 필요한 콘텐츠를 충분히 발견하거나 연결할 상품 자체가 부족함
- 추가 확인 조건: 기준 콘텐츠 목록이나 참여자 조건이 불명확함
- 기준 성격: `잠정 기준`
- 기획자가 실행 전에 정할 값: 참여자 수, 제한 시간, 필수 발견 항목과 최소 차이

### 결과에 따른 다음 선택

| 결과 | 준비 상태 변화 | 다음 행동 |
|---|---|---|
| 탐색 문제가 관찰됨 | `REFINE` | 도시×테마 하나의 최소 시안 제작 |

일반 추천:

> 축구를 좋아함 → 축구 여행 추천

> 축구·야구를 좋아함 → 스포츠 여행을 후보에서 제외 → 남은 경험 중 하나를 복불복으로 선택

즉, AI가 새로운 취향을 추측하지 않아. 사용자가 이미 알고 있는 취향만 제거하고, 남은 미지의 영역에서 여행을 뽑는 거야.

구조는 이렇게 돼.

```text
실제로 예약 가능한 여행 후보
- 평소 좋아하는 것
- 이미 경험한 것
- 절대 싫거나 불가능한 것
= 복불복 여행 후보군
```

여기서 질문도 세 종류로 분리해야 해.

- 좋아하는 것: 새로운 경험을 위해 이번에는 제외
- 이미 해본 것: 반복 방지를 위해 제외
- 절대 피해야 하는 것: 안전·건강·개인 사정 때문에 제외

예를 들면:

- 좋아함: 축구, 야구
- 경험함: 일본, 태국
- 절대 불가: 고소공포증, 날음식
- 결과 후보: 애니메이션 성지, 도예 체험, 건축 여행, 재즈 공연 등

핵심 문장은 이거야.

> **좋아하는 것을 더 추천하는 대신, 이미 좋아하는 것을 잠시 제외해 아직 모르는 나를 만나는 여행**

여행을 다녀와 새롭게 좋아하게 된 경험은 다음 복불복에서 다시 제외 목록에 들어가. 사용할수록 추천 범위가 좁아지는 게 아니라, 사용자가 계속 새로운 영역으로 밀려나는 구조가 되는 거지.

이제 이 아이디어는 단순 랜덤 여행보다 훨씬 명확해졌어. 이름을 붙이면 `역취향 추천`, `안티 추천`, 혹은 **“내가 아는 나를 뺀 여행”**에 가까워.

## Codex

맞아. **지금 v0.1은 REST API `/v1/products/flight/...`를 직접 연결하지 않았어.** 대신 마이리얼트립 공식 MCP 서버를 연결했고, 그 안의 `getCategoryList`, `searchTnas`로 실제 투어 데이터를 조회하도록 만들었어.

현재 범위가:

> 여행지 + 취미 → 해당 여행지의 낯선 투어 추천

이므로 항공권 API는 필요하지 않아. 공항 자동완성과 최저가 캘린더를 넣으면 다시 항공·목적지 추천까지 범위가 커져.

정리하면:

- 공식 투어 데이터 연결: 완료(MCP)
- 실제 오사카 상품·가격·링크 조회: 확인
- 항공 REST API 직접 연결: 미구현이며 현재 MVP 범위 밖
- 직접 API 방식으로 바꾸려면 API 키가 필요하며, 키는 로그에 남기지 말고 환경변수로 연결해야 함

따라서 지금 문제는 “공식 데이터를 안 썼다”가 아니라 **공식 REST API 대신 공식 MCP를 사용했다**가 정확해. 이번 플러그인에는 MCP 방식이 더 간결하고 과제 취지에도 맞아.

## Codex

맞아. 다만 **API를 쓴다는 사실만으로 해결되는 건 아니고, 구조화된 응답 필드로 지역 일치 검사를 해야** 오추천을 막을 수 있어.

MCP 결과에서는 숨겨졌던 필드가 REST API에는 있어:

- `description`: `도쿄 ∙ 투어`처럼 지역 확인
- `category`: 교통·티켓·투어 구분
- `gid`: 상세 정보 재조회
- `productUrl`: 실제 상품 링크
- `itemName`: 다른 도시명이 섞였는지 확인

따라서 흐름을 이렇게 바꾸는 게 정확해.

1. `/v1/products/tna/categories`로 도시 카테고리 조회
2. `/v1/products/tna/search`로 상품 조회
3. `description`의 지역이 입력 여행지와 일치하는 상품만 유지
4. 제목·설명에 다른 도시가 명시되면 제외
5. 이동·유심 같은 비체험 카테고리 제외
6. 취미와 직접 겹치는 상품 제외
7. 선택 후보를 `gid`로 상세 조회
8. 검증된 `productUrl`만 제시

즉 도쿄 검색에서 파리 미술관이 나온 경우:

> 검색어는 도쿄지만 제목에 파리가 명시됨 → 지역 충돌 → 즉시 제외

이 방식이 MCP 화면 결과만 분류하는 것보다 훨씬 안정적이야. API 키는 `MYREALTRIP_API_KEY` 같은 환경변수로만 받고, 코드·대화·로그에는 절대 남기지 않으면 돼. 다음 버전은 **공식 REST API + 지역 일치 필터**로 전환하는 게 맞겠다.
