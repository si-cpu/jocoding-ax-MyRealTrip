# Project Charter: T&A Experience Stamp Map

- 문서 버전: 0.1.0
- 상태: 초안
- 작성일: 2026-07-16
- 대상 기업/직무 맥락: 마이리얼트립 T&A실 해외 사업 개발 매니저
- 제품 형태: 웹 기반 MVP 포트폴리오
- 출발점: 기존 `여행 경험 진열대` Codex 플러그인의 제품화 확장
- 한 줄 정의: 한국 5개 도시와 일본 4개 지역의 T&A 경험을 상품 데이터와 지역 공식 자산으로 재구성하고, 여행자에게는 경험 스탬프와 공유 가능한 여행 스토리를 제공하며, 사용자가 남긴 피드백과 단계별 이벤트 데이터베이스를 통해 대표 경험·숨은 선택지·상품화 후보를 분석할 수 있게 하는 웹 서비스

## 1. 프로젝트 배경

기존 `여행 경험 진열대` 플러그인은 여행지를 입력하면 마이리얼트립 T&A 상품 안에 흩어진 장소·음식·액티비티·공연·작품·팀 같은 구체적 경험을 추출해, 상품 목록보다 먼저 경험 지도를 보여주는 도구였다. 이 프로젝트는 그 아이디어를 한 단계 확장한다.

여행자는 단순히 상품을 예약하는 데서 끝나지 않는다. 여행 전에는 “이 도시에서 무엇을 할 수 있는가”를 파악하고, 여행 중에는 실제로 한 경험을 사진과 메모로 기록하며, 여행 후에는 그 기록을 나다운 여행 이야기로 남긴다. 반면 사업개발자는 도시별로 어떤 경험이 이미 상품화되어 있고, 어떤 지역 자산이 아직 상품화되지 않았는지 파악해야 신규 카테고리 발굴, 파트너 제안, 프로모션 기획을 할 수 있다.

따라서 본 프로젝트는 T&A 상품 탐색을 `발견 → 선택 → 기록 → 스토리화 → 공유·피드백`의 여행자 흐름으로 확장하고, 각 단계에서 남는 비식별·집계 데이터를 구조화해 사업개발 인사이트의 기반으로 삼는다. 단, 일반 사용자에게 내부 리포트나 사업개발 분석 화면을 직접 노출하지 않으며, 내부 분석자는 사진 원본·원문 메모·개별 사용자의 여행 스토리를 열람하지 않는다.

## 2. 문제 정의

### 2.1 여행자 문제

여행자는 목적지를 정해도 그 도시에서 할 수 있는 경험을 한눈에 파악하기 어렵다. 기존 상품 목록은 투어·입장권·액티비티·미식 같은 판매 형식 중심으로 구성되어 있어, 사용자가 도시의 매력을 경험 단위로 이해하려면 여러 상품 상세를 반복해서 열어봐야 한다. 또한 여행이 끝난 뒤에는 사진이 갤러리에 흩어지고, 어떤 경험을 왜 선택했는지 스토리로 남기기 어렵다.

### 2.2 사업개발 문제

T&A 사업개발자는 신규 지역·카테고리·파트너 기회를 발굴해야 한다. 하지만 상품화된 경험과 지역이 공식적으로 밀고 있는 먹거리·관광지·인증 장소·축제·로컬 자산이 분리되어 있고, 여행자가 실제로 어떤 경험을 선택·기록·스토리화했는지 단계별 데이터가 남지 않으면 어떤 경험이 이미 검증된 대표 콘텐츠이고 어떤 경험이 상품화 후보인지 빠르게 판단하기 어렵다.

## 3. 해결 개념

`도시 선택 → T&A 경험 지도 → 경험 선택 → 스탬프 보드 → 사진/메모 기록 → 공유 가능한 여행 스토리 생성 → 피드백 입력 → 비식별 통계 축적 → 내부 사업개발 분석`

여행자에게는 상품을 먼저 추천하지 않고 도시별 경험 지도를 먼저 보여준다. 사용자가 이번 여행에 담고 싶은 경험을 선택하면 스탬프 보드가 만들어지고, 여행 중 각 경험에 사진과 한 줄 메모를 붙일 수 있다. 여행 종료 후에는 선택한 경험, 사진 유형, 메모, 방문 순서를 바탕으로 개인/단체 여행에 맞는 공유 가능한 여행 여정 스토리를 생성한다. 사용자는 스토리를 확인한 뒤 공개 여부를 선택하고, 각 경험에 대한 간단한 피드백을 남길 수 있다.

사업개발 인사이트는 일반 사용자 화면이 아니라 내부/운영자용 분석으로 둔다. 내부 분석은 상품 노출 수, 경험 선택 수, 스탬프 수, 사진 유형 분포, 스토리 생성·공유 수, 만족도·추천 의향 같은 집계 통계만 사용한다. 특히 각 단계의 완료율과 전환율을 핵심 지표로 삼아, 사용자가 어디서 멈추는지와 어떤 경험이 실제 여행 스토리까지 이어지는지 확인한다. 또한 도시별 조회량, 여행 계획 포기 사유, 교통·숙소·액티비티 등 비용 카테고리별 예상 지출을 함께 보면 지역별 수요와 가격 민감도를 더 구체적으로 파악할 수 있다. 사진 원본, 원문 메모, 원문 피드백, 개별 여행 스토리는 개인정보와 초상권 리스크가 있으므로 내부 리포트에 저장하거나 노출하지 않는다. 이 집계 데이터가 쌓이면 마이리얼트립 상품 근거가 많은 경험은 `대표 경험` 또는 `많이 검증된 경험`으로, 상품 근거는 적지만 지역 공식 자산과 사용자 기록·피드백 신호가 강한 항목은 `상품화 후보` 또는 `파트너 발굴 후보`로 분석할 수 있다.

## 4. MVP 범위

### 4.1 대상 도시

MVP는 한국 5개 도시와 일본 4개 지역을 대상으로 한다.

#### 한국

- 서울
- 전주
- 여수
- 부산
- 대구

#### 일본

- 도쿄
- 오사카
- 홋카이도
- 후쿠오카

이 도시 세트는 단순 광역시 기준이 아니라 T&A 사업개발 검증용 포트폴리오다. 오사카처럼 공급이 풍부한 도시, 전주·여수처럼 특정 테마가 강한 도시, 대구처럼 상품화 후보를 찾기 좋은 도시, 홋카이도처럼 계절·자연·근교형 경험이 강한 지역을 함께 비교한다.

### 4.2 도시별 역할

| 도시 | 역할 | 검증 포인트 |
|---|---|---|
| 서울 | 복합 도시형 | 공연·미식·스냅·클래스·야간 콘텐츠 |
| 전주 | 전통·미식형 | 한옥·한식·전통 체험·스냅 |
| 여수 | 해양·야경형 | 바다·케이블카·야경·로컬 미식 |
| 부산 | 국내 대형 관광형 | 해양·액티비티·야경·미식·스냅 |
| 대구 | 발굴 후보형 | 로컬 미식·근대골목·시장·야경 상품화 후보 |
| 도쿄 | 초대형 해외 도시형 | 엔터테인먼트·전망대·테마파크·미식 |
| 오사카 | 고밀도 T&A 도시형 | USJ·전망대·패스·미식·근교투어 |
| 홋카이도 | 자연·계절·근교형 | 비에이·후라노·눈·온천·계절 투어 |
| 후쿠오카 | 근거리 일본 도시형 | 미식·근교·온천·쇼핑·도시 경험 |

### 4.3 MVP 기능

1. 도시 선택 화면
2. 도시별 경험 지도
3. 경험별 상태 구분
   - 예약 가능 경험
   - 지역 공식 자산
   - 상품화 후보
4. 경험 카드
   - 테마
   - 근거 유형
   - 연결 상품 수
   - 지역 공식 자산 여부
   - 구매 방식
   - 사용자용 상태·근거 요약
5. 스탬프 보드
   - 사용자가 담은 경험 목록
   - 스탬프 획득 상태
   - 개인/단체 여행 구분
6. 사진/메모 기록
   - 사진 업로드 또는 샘플 이미지 선택
   - 개인 사진/단체 사진/풍경/음식 등 사진 유형 선택
   - 한 줄 메모
7. 여행 스토리 생성
   - 개인 여행용 문체
   - 단체 여행용 문체
   - 경험 테마 기반 요약
   - 공유 가능한 최종 여행 여정 형태
8. 단계별 데이터 기록
   - 도시 조회
   - 경험 노출
   - 경험 선택
   - 구매 방식 확인
   - 스탬프 추가
   - 사진 유형 선택
   - 메모 작성
   - 스토리 생성
   - 스토리 공유 여부
   - 경험별 피드백
   - 단계별 완료 여부
   - 단계별 실패·오류
   - 여행 포기/보류 사유
   - 비용 추정 입력 또는 산출
9. 내부 데이터 기반 분석 리포트
   - 대표 경험
   - 숨은 선택지
   - 상품화 부족 영역
   - 신규 파트너 발굴 후보
   - 프로모션/CRM/SNS 소재 후보
   - 단계별 완료율·전환율
   - 오류 유형별 개선 포인트
   - 도시별 조회량
   - 여행 포기/보류 사유 분포
   - 교통·숙소·액티비티 예상 비용 분포

## 5. 핵심 사용자

### 5.1 여행자

- 목적지는 정했지만 무엇을 할지 아직 정하지 못한 사용자
- 상품 목록보다 도시의 매력을 먼저 보고 싶은 사용자
- 여행 중 경험을 기록하고 여행 후 스토리로 남기고 싶은 사용자
- 개인 여행과 단체 여행의 기록 방식을 다르게 관리하고 싶은 사용자

### 5.2 사업개발자/콘텐츠 운영자

- 도시별 T&A 상품 기회를 찾는 사업개발자
- 기존 상품의 대표 경험과 덜 노출된 경험을 구분해야 하는 담당자
- 지역 공식 자산, 상품 공급, 사용자 선택·기록·피드백 데이터를 함께 보고 신규 파트너를 발굴하려는 담당자
- CRM, SNS, 상세페이지, 프로모션 소재를 기획하는 콘텐츠 담당자

## 6. 데이터 전략

### 6.1 데이터 유형

| 데이터 유형 | 의미 | 사용 방식 |
|---|---|---|
| 마이리얼트립 T&A 상품 데이터 | 이미 상품화된 경험 | 예약 가능 경험, 구매 방식, 대표 경험 산정 |
| 지역 공식 자산 | 지자체·관광공사·공식/인증 출처의 먹거리·명소·축제·인증 장소 | 상품화 후보, 공급 갭 분석 |
| 단계별 행동 데이터 | 도시 조회, 경험 노출, 경험 선택, 구매 방식 확인, 스탬프 기록, 스토리 생성 | 단계별 완료율, 전환율, 이탈 구간, 상품화 후보 분석 |
| 오류 로그 데이터 | 데이터 없음, 상품 링크 실패, 스토리 생성 실패, 피드백 저장 실패 등 | 운영 개선, 데이터 보강, UI 개선 우선순위 |
| 사용자 스탬프 데이터 | 사용자가 선택·기록한 경험 | 여행자 본인의 스토리 생성 |
| 메모/사진 유형 | 사용자가 직접 입력·선택한 기록 | 여행자 본인의 스토리 문체 구성. 내부 분석에는 사진 원본·원문 메모를 사용하지 않음 |
| 사용자 피드백 집계 데이터 | 경험 만족도, 재방문 의향, 추천 의향, 불편 유형의 통계 | 내부 상품 개선, 파트너 제안, 콘텐츠 보완 |
| 여행 포기/보류 사유 데이터 | 가격 부담, 일정 부족, 교통 불편, 동행자 미확정, 상품 부족 등 선택형 사유 | 전환 저해 요인, 상품/가격/콘텐츠 개선 |
| 비용 추정 데이터 | 교통비, 숙소비, 액티비티 비용, 기타 비용의 추정값 | 도시별 예산 감도, 패키지·프로모션 가격대 설계 |
| 예약·방문 의도 신호 | 저장, 여행 보드 추가, 구매 방식 조회, 상품 URL 열람, 날짜 입력, 방문 완료 표시 | 관심·계획·구매 검토·방문 완료 단계 추정 |

### 6.2 근거 라벨

모든 경험은 다음 중 하나 이상의 근거 라벨을 가진다.

- `MRT_PRODUCT`: 마이리얼트립 상품 근거 있음
- `OFFICIAL_CITY_ASSET`: 도시·지자체·관광공사 공식 자산 근거 있음
- `CERTIFIED_FOOD`: 인증 식당·대표 먹거리 근거 있음
- `CERTIFIED_PLACE`: 인증/추천 관광지 근거 있음
- `USER_STAMP`: 사용자가 여행 중 기록한 경험
- `HYPOTHESIS`: 상품화 후보로 제안된 가설

| 지역 공식 자산 | 지역 공식·인증 출처 근거는 있으나 상품 연결이 없거나 약한 경험 | 지역 자산 |
| 상품화 후보 | 지역 자산 또는 사용자 기록 가능성은 있으나 상품 공급이 부족한 경험 | 파트너 발굴 후보 |
| 근거 부족 | 공식 상품·지역 자산 근거가 부족한 항목 | 표시 제외 또는 검증 필요 |

## 10. 성공 기준

### MVP 산출 기준

- 9개 도시가 동일 데이터 구조로 등록되어 있다.
- 각 도시는 최소 3개 이상의 경험 또는 지역 자산을 가진다.
- 최소 2개 도시는 스탬프 보드와 스토리 생성 데모를 제공한다.
- 최소 1개 도시는 단계별 이벤트 로그 기반 공급 갭 분석을 내부 리포트 형태로 보여준다.
- 도시 조회, 경험 노출, 경험 선택, 스탬프 기록, 스토리 생성, 공유, 피드백 이벤트 구조가 정의되어 있다.
- 단계별 완료율과 이탈률을 계산할 수 있다.
- 실패·오류 로그 구조가 정의되어 있다.
- 도시별 조회량과 여행 보류/포기 사유를 집계할 수 있다.
- 교통·숙소·액티비티 비용 추정 구조가 정의되어 있다.
- 예약 가능 경험과 상품화 후보를 명확히 구분한다.
- 공식 상품 근거가 없는 항목을 예약 가능으로 표현하지 않는다.

### 사용자 가치 기준

- 사용자가 도시별 주요 경험을 상품 목록 없이 파악할 수 있다.
- 사용자가 이번 여행에 담을 경험을 직접 선택할 수 있다.
- 사용자가 여행 중 기록을 남기고 여행 후 공유 가능한 여행 여정 스토리로 확인할 수 있다.
- 사용자가 각 경험에 대한 간단한 피드백을 남길 수 있다.

### 사업개발 가치 기준

- 도시별 대표 경험과 숨은 선택지를 구분한다.
- 지역 공식 자산 대비 상품 공급이 부족한 영역을 찾는다.
- 단계별 행동 데이터와 피드백 데이터로 파트너 발굴·프로모션·콘텐츠 소재 후보를 제안한다.
- 단계별 완료율과 오류 로그로 운영·콘텐츠·데이터 보강 우선순위를 정한다.
- 도시별 조회량과 포기 사유로 수요는 있으나 전환되지 않는 지역을 찾는다.
- 비용 카테고리별 추정값으로 가격대·패키지·프로모션 설계 가설을 만든다.
- 내부 리포트는 일반 사용자에게 노출하지 않는다.

## 11. 부적절한 처리 및 안전 원칙

- 공식 상품 URL이 없는 경험을 예약 가능하다고 표시하지 않는다.
- 지역 공식 자산을 마이리얼트립 판매 상품처럼 표현하지 않는다.
- 사용자 사진에서 얼굴·신원·관계·국적을 추론하지 않는다.
- 개인/단체 여부는 사용자가 직접 선택한 값만 사용한다.
- 사진과 메모는 사용자가 직접 업로드·입력한 경우에만 사용한다.
- 행동 데이터는 MVP에서 익명 세션 단위로만 기록하고 개인 식별 정보와 실제 위치는 저장하지 않는다.
- 내부 분석 DB에는 사진 원본, 원문 메모, 원문 피드백, 개별 여행 스토리를 저장하거나 노출하지 않는다.
- 내부 분석에는 선택형 피드백, 사진 유형, 스토리 생성 여부, 공유 여부, 경험별 선택·스탬프 수 같은 집계 통계만 사용한다.
- 오류 로그에도 개인 식별 정보, 원문 메모, 사진 원본, 개별 여행 스토리를 저장하지 않는다.
- 스토리 공유는 사용자의 확인 이후에만 가능하다고 설계한다.
- 피드백은 집계 분석에 사용될 수 있음을 명시하고, 개인 식별 정보와 결합하지 않는다.
- 여행 포기/보류 사유는 선택형 카테고리로만 저장하고 자유서술 원문은 내부 분석 DB에 저장하지 않는다.
- 비용 추정은 사용자가 직접 입력한 값, 공개 평균, 공식 상품 가격 기반 추정값만 사용하며 카드 내역·계좌 정보·개인 금융 정보는 사용하지 않는다.
- 예약·방문 의도는 저장, 여행 보드 추가, 구매 방식 조회, 상품 URL 열람, 날짜 입력, 방문 완료 표시처럼 사용자가 직접 남긴 서비스 행동만으로 추정한다.
- 식비는 총액 또는 예산 범위만 선택적으로 다루며, 사용자가 무엇을 먹었는지 추론하지 않는다.
- 내부 데이터 리포트는 일반 사용자 화면에 노출하지 않는다.
- 상품 겹침 수를 실제 방문객 수, 한국인 비율, 한적함으로 해석하지 않는다.
- 모텔·펜션 등 숙박 유형에 대한 사회적 편견 표현은 본 프로젝트 범위에서 사용하지 않는다.

## 17. 운영 계획

### 17.1 데이터 운영

- 9개 도시의 샘플 데이터는 `도시`, `경험`, `상품 링크`, `지역 공식 자산`, `상태`, `출처`, `수집일`을 분리해 관리한다.
- 지역 공식 자산은 지자체, 관광공사, 공식 인증 자료 등 공개 출처만 사용한다.
- 마이리얼트립 상품 근거가 없는 항목은 `상품화 후보` 또는 `지역 공식 자산`으로만 표시한다.
- 도시별 데이터 밀도는 `high`, `medium`, `low`, `gap_candidate`로 관리한다.
- 비용 추정은 실제 결제액이 아니라 추정값임을 모든 화면과 리포트에 표시한다.

### 17.2 이벤트/로그 운영

- 모든 주요 사용자 흐름은 `event_logs`로 기록한다.
- 집계 지표는 `funnel_metrics`로 계산한다.
- 오류는 `error_logs`에 남기고, 오류 유형·발생 단계·심각도를 구분한다.
- 원문 메모, 사진 원본, 원문 피드백, 개별 여행 스토리는 내부 분석 로그에 남기지 않는다.
- 로그는 개인 식별 정보와 결합하지 않고 익명 세션 단위로만 다룬다.

### 17.3 리포트 운영

- 내부 리포트는 일반 사용자에게 노출하지 않는다.
- 내부 리포트는 비식별 집계값만 사용한다.
- 도시별 조회량, 경험 선택률, 스탬프 완료율, 스토리 생성률, 공유율, 피드백률을 기본 지표로 둔다.
- 포기/보류 사유와 비용 추정은 전환 저해 요인을 찾는 보조 지표로만 사용한다.
- 리포트는 상품개발·파트너 발굴·프로모션·콘텐츠 개선 가설을 제시하되, 실제 매출 개선을 단정하지 않는다.

### 19.4 완료 조건

- 9개 도시가 화면에 표시된다.
- 각 도시가 최소 3개 이상의 경험 또는 공식 자산을 가진다.
- 최소 2개 도시에서 스탬프 보드와 스토리 생성이 가능하다.
- 단계별 이벤트가 localStorage 또는 DB에 기록된다.
- 완료율과 이탈률이 계산된다.
- 오류 로그가 남는다.
- 포기/보류 사유와 비용 추정 입력이 가능하다.
- 내부 리포트가 비식별 집계값만 보여준다.
