# Data Exploration: 여행 콘텐츠 키워드 지도

- 버전: 0.1
- 작성일: 2026-07-06
- 목적: Project Charter 0.6의 데이터 실행 가능성과 키워드 추출 경계를 확인한다.

## 1. 결론

마이리얼트립 TNA API만으로 MVP 구현이 가능하다.

- 검색 API: 도시별 상품 수집, 상품명, 상품 유형, 가격, 공식 URL 확보
- 상세 API: 상품 소개, 포함·불포함 사항, 일정 제목·설명 확보
- 옵션 API: 사용자가 날짜를 입력한 경우 해당 날짜의 구매 옵션 확인

키워드는 검색 결과의 상품명만으로 만들지 않는다. 상품 안에 숨은 장소·음식·작품을 찾으려면 상세 API의 `description`과 `itineraries`가 필요하다.

단, API 키가 현재 로컬 환경에 설정되어 있지 않아 이번 단계에서는 공식 문서의 응답 스키마와 공개 상품 페이지로 데이터 가능성을 확인했다. 실제 응답 표본 저장은 키가 설정된 실행 환경에서 추가해야 한다.

## 2. 공식 API 데이터 경계

### 2.1 상품 검색

`POST /v1/products/tna/search`

주요 입력:

- `keyword`: 필수, 도시명 또는 상품명
- `category`: 선택, 도시별 카테고리 API 반환값만 사용
- `sort`: 가격·리뷰·판매량 기준
- `page`: 1부터 시작
- `size`: 최대 100

키워드 지도에 사용할 응답:

| 필드 | 용도 |
|---|---|
| `gid` | 상세·옵션 조회 및 상품 식별 |
| `itemName` | 1차 콘텐츠 후보 추출 |
| `description` | 도시와 상품 유형 검증 |
| `category` | 교통·유심 등 비대상 상품 제외 |
| `productUrl` | 공식 상품 페이지 연결 |
| `salePrice`, `priceDisplay` | 시작가 표시가 필요할 때 사용 |
| `hasNextPage` | 전체 페이지 순회 |

검색 API의 `description`은 예시상 `오사카 ∙ 투어` 형태이므로 상품 내용을 추출하는 본문으로 사용하지 않는다.

### 2.2 상품 상세

`POST /v1/products/tna/detail`

| 필드 | 용도 |
|---|---|
| `title` | 검색 상품명 교차 확인 |
| `description` | 장소·작품·음식·활동 후보 추출 |
| `included` | 실제 포함 경험과 단순 소개 구분 |
| `excluded` | 식사·입장권 등이 결제 대상이 아닌지 확인 |
| `itineraries[].title` | 방문 장소·행동 추출의 주요 근거 |
| `itineraries[].description` | 일정별 세부 콘텐츠 근거 |

일부 연동 상품은 상세 정보가 없을 수 있으므로 검색 결과만으로 확인 가능한 키워드만 허용하는 fallback이 필요하다.

### 2.3 날짜별 옵션

`POST /v1/products/tna/options`

- 날짜 입력이 있을 때만 호출한다.
- 옵션 목록이 존재해야 해당 날짜에 구매 가능한 후보로 표시한다.
- 검색 API의 시작가만으로 선택 날짜의 예약 가능성을 주장하지 않는다.

### 표본 판단

1. 도시 규모와 상품 수가 달라도 장소·시설·음식 단위의 키워드 추출은 가능하다.
2. 하나의 상품에서 여러 콘텐츠 키워드를 추출할 수 있다.
3. 먹거리 상품 연결성은 도시 편차가 크다.
4. 유명 음식이라는 외부 상식만으로는 마이리얼트립 예약 상품에 연결할 수 없다.
5. 지역 입력은 실제 상품 도시로 확장해야 할 수 있다. 예: `연해주 → 블라디보스토크·우수리스크`. 이 확장은 사용자에게 명시해야 한다.

## 4. 수집 전략

### 4.1 기본 순서

1. 도시명을 `keyword`로 TNA 검색한다.
2. `hasNextPage`가 거짓이 될 때까지 최대 100개씩 순회한다.
3. `description`의 도시가 요청 도시와 일치하는 상품만 남긴다.
4. 이동·교통, 유심, 대여 등 경험 발견 목적과 맞지 않는 카테고리를 제외한다.
5. `gid`를 기준으로 중복 상품을 제거한다.
6. 남은 상품의 상세 정보를 조회한다.
7. HTML을 제거하고 제목·소개·포함사항·일정에서 후보를 추출한다.
8. 후보마다 원문 필드와 `gid`를 근거로 보존한다.
9. 동의어를 통합하고 `뭐 하지?`와 `뭐 먹지?`로 나눈다.
10. 키워드별 공식 `productUrl` 목록을 출력한다.
11. 실제 노출 직전에 HTTPS, 공식 도메인, HTTP 응답과 리디렉션 최종 도메인을 확인한다.

### 4.2 호출량과 지연 대응

검색은 분당 최대 200건, 상세도 분당 최대 200건이다. 상품이 많은 도시에서 모든 상세를 대화마다 다시 조회하면 지연과 불필요한 호출이 발생한다.

MVP 대응:

- 같은 실행 안에서 `gid`별 상세 응답을 메모리 캐시한다.
- 도시별 검색 결과를 한 번 수집한 뒤 재사용한다.
- 상세 조회 실패 상품은 제목에서 직접 확인되는 키워드만 사용한다.
- 상품 전체를 보았다고 주장하려면 모든 페이지 순회를 완료해야 한다.
- 페이지나 상세 일부만 조회했다면 `확인된 상품 범위에서`라고 표시한다.

향후 운영 대응:

- 주기적 사전 색인과 변경 상품 증분 갱신
- 도시별 키워드 캐시
- 품절·판매 중지 상품의 만료 처리

## 5. 키워드 근거 모델

각 키워드는 최소 다음 구조를 가진다.

```json
{
  "keyword": "슈퍼 닌텐도 월드",
  "type": "DO",
  "normalizedKey": "슈퍼 닌텐도 월드",
  "evidence": [
    {
      "gid": "상품 ID",
      "sourceField": "description",
      "sourceText": "상품에 실제로 등장한 근거 문구",
      "productUrl": "공식 상품 URL"
    }
  ]
}
```

`type` 후보:

- `DO`: 장소, 시설, 작품, 캐릭터, 팀, 공연, 활동
- `EAT`: 음식, 식당 예약, 식사 체험, 푸드투어

키워드와 상품의 관계는 다음 셋 중 하나로 구분한다.

- `DIRECT`: 상품에 동일 표현이 직접 등장
- `ALIAS`: 검증된 동의어 또는 공식 약칭
- `UNSUPPORTED`: 연상은 가능하지만 직접 근거 없음 — 출력 제외

## 6. 다음 구현 결정

1. 기존 항공·숙소 명령을 코드와 스킬에서 제거한다.
2. TNA 검색·상세·옵션만 남긴 전용 클라이언트로 단순화한다.
3. 전체 페이지 수집 명령과 상세 일괄 조회 기능을 추가한다.
4. 키워드 후보 생성과 최종 의미 판단을 분리한다.
5. 모든 출력 키워드에 `gid`, 근거 필드, 공식 URL을 추적한다.
6. 먹거리는 예약 가능한 상품 근거가 있는 항목을 우선하며 정보성 항목을 섞지 않는다.

## 7. 미해결 항목

- 실제 API 응답에서 도시명 검색이 반환하는 상품 범위와 도시 필터 정확도
- 상세 API의 HTML 품질과 긴 설명 처리 방식
- 판매 중지 상품이 검색 결과에 남는지 여부
- 도시별 먹거리 상품 연결 비율
- 적정 키워드 개수와 묶음 방식
- API 키가 없는 사용자에게 제공할 실패 경험

이 항목들은 실제 API 키를 환경변수로 설정한 뒤 표본 수집과 테스트에서 확인한다. 키는 대화나 로그에 입력하지 않는다.
