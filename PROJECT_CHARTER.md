# Project Charter

## 1. 프로젝트 개요

### 프로젝트명

**MRT Idea Gate** *(가칭)*

### 한 줄 정의

마이리얼트립의 서비스·콘텐츠 기획자가 개인적으로 떠올린 아이디어가 정식 기획으로 발전시킬 가치가 있는지 가장 먼저 확인하도록 돕는 Codex 플러그인이다.

### 핵심 판단 질문

> 이 개인적인 아이디어는 시간과 노력을 들여 정식 기획으로 발전시킬 가치가 있는가?

### 적용 단계

본 플러그인은 아이디어와 기획 사이의 가장 첫 번째 확인 단계에만 적용한다.

```text
개인이 떠올린 아이디어
→ Idea Gate 최초 확인
→ 기획으로 발전시킬 가치 판단
→ 가치가 있을 때 정식 기획 시작
```

아직 팀의 기획 과제로 채택되지 않았고 구체적인 요구사항이나 실행 계획도 없는 상태에서 사용한다. 이미 정식 기획이 시작된 대상을 평가하는 도구가 아니다.

### 추진 배경

마이리얼트립은 AI Native 조직으로 전환하며 구성원 한 명이 문제를 발견하고 직접 해결까지 이어가는 업무 방식을 확대하고 있다. 개인이 아이디어를 더 쉽게 실행할 수 있는 환경일수록, 실행에 앞서 그 아이디어가 기획할 가치가 있는지 확인하는 초기 판단이 중요해진다.

개인적인 아이디어에는 개인 경험, 관찰한 불편, 떠오른 해결책과 기대 효과가 섞이기 쉽다. 이 상태에서 바로 정식 기획을 시작하면 아이디어의 실제 가치보다 기획 문서를 만드는 일에 먼저 시간과 노력을 쓰게 될 수 있다.

본 프로젝트는 아이디어를 더 그럴듯하게 문서화하는 도구가 아니라, 개인적인 생각을 정식 기획으로 발전시킬 가치가 있는지 가장 초기에 확인하는 의사결정 보조 시스템을 만드는 것을 목적으로 한다.

### 대상 사용자

마이리얼트립에서 서비스 또는 콘텐츠와 관련된 개인적인 아이디어를 떠올리고, 이를 정식 기획으로 발전시킬지 고민하는 **서비스 기획자**와 **콘텐츠 기획자**를 주 사용자로 한다.

### 프로젝트 목적

- 초기 아이디어에서 문제와 해결책을 분리한다.
- 중요 주장을 사실, 개인 경험, 추론, 미확인 정보로 구분한다.
- 기존 기능과 공개 근거를 확인한다.
- 가장 강한 반례와 더 작은 대안을 검토한다.
- 검증이 필요한 항목과 다음 행동을 명확히 한다.
- 근거 상태에 따라 `GO`, `REVISE`, `HOLD`, `DROP` 판단을 지원한다.

### 기대 결과

사용자가 개인적인 아이디어 한 건을 입력하면, Codex가 그 아이디어를 정식 기획으로 발전시킬 가치가 있는지 확인한 Idea Gate 결과를 생성한다. 가치가 있다고 판단된 아이디어는 정식 기획을 시작할 후보가 되며, 보완·보류·중단을 권고하는 결과도 정상적인 성과로 취급한다.

### 현재 단계

프로젝트 방향과 대상 사용자를 확정하고 Project Charter를 단계적으로 작성하는 중이다. 산출물, 성공 기준과 세부 요구사항은 후속 섹션에서 정의한다.

## 2. 프로젝트 범위

### 대상 업무

서비스 기획자와 콘텐츠 기획자가 개인적으로 떠올린 아이디어를 정식 기획 과제로 제안하기 전에 수행하는 최초 확인 단계만 다룬다.

- 서비스 기획: 신규 기능, 사용자 흐름 개선, 예약·결제·상품 탐색 경험, 운영 자동화 아이디어
- 콘텐츠 기획: 여행 테마, 상품 큐레이션, 프로모션, 에디토리얼·캠페인 콘텐츠 아이디어

### 공통 범위

- 한 번에 하나의 아이디어를 검토한다.
- 아이디어에서 대상 사용자, 문제, 제안 해결책을 분리한다.
- 중요 주장을 `FACT`, `EXPERIENCE`, `INFERENCE`, `UNKNOWN`으로 구분한다.
- 입력된 자료와 공개적으로 검증 가능한 자료를 근거로 사용한다.
- 기존 기능, 기존 콘텐츠, 유사 사례와 중복 가능성을 확인한다.
- 가장 강한 반례, 실패 조건과 더 작은 대안을 제시한다.
- 검증되지 않은 가정과 다음 검증 행동을 정리한다.
- 역할별 체크리스트와 공통 게이트 규칙을 적용한다.
- 결과를 `GO`, `REVISE`, `HOLD`, `DROP` 중 하나로 제안하고 근거를 남긴다.

### 서비스 기획 검토 범위

- 문제 발생 사용자와 사용자 여정
- 기존 기능 및 우회 행동
- 기능 추가 필요성과 더 작은 해결책
- 필요한 데이터·시스템 의존성
- 예외 상황, 오작동과 운영 위험
- 측정 가능한 사용자·사업 결과

### 콘텐츠 기획 검토 범위

- 대상 여행자와 콘텐츠 소비 상황
- 기존 콘텐츠·테마·프로모션과의 중복
- 정보 출처, 최신성, 정확성과 저작권 위험
- 시즌·지역·상품 공급 조건과 실행 가능성
- 노출 채널과 기대 행동
- 측정 가능한 콘텐츠 성과

### 범위 밖

본 단계에서는 Idea Gate 검토 결과를 생성하는 범위까지만 정의한다. 검토 이후의 정식 기획과 실행 과정은 본 프로젝트의 현재 범위에서 다루지 않는다.

### 단계 종료 조건

플러그인의 책임은 개인적인 아이디어를 정식 기획으로 발전시킬 가치가 있는지 확인 결과를 제공하는 시점에 끝난다. `GO`는 정식 기획을 시작해볼 가치가 있다는 권고이며, 최종 판단과 책임은 사용자에게 있다.

## 3. 핵심 사용자 흐름

### 사용 시작 조건

사용자가 서비스 또는 콘텐츠와 관련된 개인적인 아이디어를 떠올렸지만, 아직 정식 기획으로 제안할지 결정하지 않은 시점에 플러그인을 사용한다.

### 최소 입력

초기 아이디어의 부담을 높이지 않도록 첫 입력은 다음 세 항목으로 제한한다.

1. 아이디어를 떠올린 사람의 역할: 서비스 기획 또는 콘텐츠 기획
2. 아이디어 한 문장
3. 아이디어를 떠올리게 된 경험이나 관찰

자료와 수치가 없어도 시작할 수 있다. 다만 개인 경험과 확인된 사실을 같은 수준의 근거로 취급하지 않는다.

### 처리 단계

#### 1단계 — 원문 보존

사용자가 입력한 아이디어와 경험을 해석하기 전 형태로 기록한다. 이후 정리 과정에서 원래 의도가 달라졌는지 비교할 수 있어야 한다.

#### 2단계 — 아이디어 구조화

입력에서 다음 항목을 분리한다.

- 예상 사용자 또는 대상
- 관찰한 문제나 기회
- 사용자가 떠올린 해결 방향
- 기대하는 변화
- 아직 확인되지 않은 가정

문제와 해결책이 섞여 있으면 이를 분리하되, 없는 내용을 임의로 보충하지 않는다.

#### 3단계 — 초기 근거 확인

입력 내용과 공개적으로 확인 가능한 자료를 기준으로 다음을 살펴본다.

- 해당 문제나 기회가 실제로 존재한다는 신호
- 마이리얼트립의 기존 서비스·콘텐츠와의 관련성
- 이미 존재하는 기능·콘텐츠·대체 방법
- 아이디어를 지지하는 근거와 약화하는 근거

자료를 찾지 못한 경우 `근거 없음`으로 단정하지 않고 `확인되지 않음`으로 표시한다.

#### 4단계 — 반례 확인

아이디어가 가치 없을 수 있는 가장 강한 이유를 최소 하나 제시한다. 반례는 억지로 반대하기 위한 문장이 아니라, 사실이라면 기획 필요성을 바꿀 수 있는 조건이어야 한다.

각 반례에는 다음을 함께 기록한다.

- 반례가 중요한 이유
- 현재 확인 상태
- 확인에 필요한 가장 작은 행동

#### 5단계 — 역할별 초기 점검

- 서비스 기획 아이디어는 사용자 흐름, 기존 우회 방법, 기능 필요성을 중심으로 점검한다.
- 콘텐츠 기획 아이디어는 대상 여행자, 콘텐츠 차별성, 시기와 상품 연결 가능성을 중심으로 점검한다.

이 단계에서는 상세 요구사항이나 실행 계획을 만들지 않는다.

#### 6단계 — 결과 제안

검토 결과를 다음 네 가지 중 하나로 제안한다.

| 결과 | 의미 |
|---|---|
| `GO` | 정식 기획으로 발전시켜 볼 초기 가치가 확인됨 |
| `REVISE` | 문제나 대상, 해결 방향을 수정하면 다시 검토할 가치가 있음 |
| `HOLD` | 핵심 정보가 부족해 작은 확인 작업이 먼저 필요함 |
| `DROP` | 현재 근거에서는 정식 기획으로 발전시킬 가치가 낮음 |

결과는 아이디어에 대한 최종 승인이나 실패 판정이 아니라, 현재 확인된 정보에 기반한 다음 행동 권고다.

### 기본 출력

플러그인은 한 아이디어에 대해 다음 내용을 간결한 검토 카드로 제공한다.

- 입력한 원문 아이디어
- 구조화된 문제와 해결 방향
- 확인된 근거와 반대 근거
- 핵심 미확인 가정
- 가장 강한 반례
- 역할별 초기 점검 결과
- `GO`, `REVISE`, `HOLD`, `DROP` 권고
- 다음으로 할 가장 작은 행동

## 4. 명제 기반 근거 판단

### 기본 원칙

플러그인은 아이디어 전체를 한 번에 평가하지 않는다. 먼저 아이디어에 포함된 주장들을 검증 가능한 명제로 분리하고, 명제별 근거 상태를 확인한 뒤 아이디어의 기획 가치를 판단한다.

### 명제 작성 조건

각 명제는 다음 조건을 만족해야 한다.

- 한 문장에는 하나의 주장만 담는다.
- 대상, 상황, 현상 또는 기대 결과가 드러나야 한다.
- 사실인지 아닌지 확인할 수 있어야 한다.
- 반대 증거가 나왔을 때 틀렸다고 판단할 수 있어야 한다.
- 근거에 없는 수량, 빈도, 원인 또는 효과를 추가하지 않는다.

좋지 않은 명제:

> 친구 여행은 결제가 불편하고 분할결제를 만들면 만족도와 매출이 높아질 것이다.

분리한 명제:

> P1. 여러 명이 하나의 마이리얼트립 상품을 예약할 때 대표자 한 명이 전체 금액을 결제하는 상황이 존재한다.

> P2. 대표 결제자는 동행자와 비용을 정산하는 과정에서 불편을 경험할 수 있다.

> P3. 이 정산 불편은 정식 기획을 검토할 만큼 반복되거나 영향이 크다.

> P4. 사용자가 현재 이용하는 송금·더치페이 방식만으로는 이 불편이 충분히 해결되지 않는다.

> P5. 마이리얼트립 내부의 분할결제 방식이 다른 대안보다 문제를 더 잘 해결할 가능성이 있다.

### 기본 명제 유형

에이전트는 아이디어에 해당하는 유형만 선택해 명제를 만든다. 모든 유형을 억지로 채우지 않는다.

| 유형 | 확인할 내용 |
|---|---|
| 대상 명제 | 문제나 기회를 경험하는 구체적인 사용자가 존재하는가 |
| 상황 명제 | 문제가 발생하는 시점과 조건이 실제로 존재하는가 |
| 문제 명제 | 관찰된 불편이나 기회가 실제 현상인가 |
| 중요성 명제 | 정식 기획을 검토할 만큼 반복성 또는 영향이 있는가 |
| 기존 대안 명제 | 현재 기능이나 우회 방법으로 충분히 해결되지 않는가 |
| 방향 명제 | 떠올린 해결 방향이 문제와 논리적으로 연결되는가 |
| 기업 적합성 명제 | 마이리얼트립이 다룰 이유와 맥락이 있는가 |

### 주장 성격 분류

아이디어 검토 과정에서 등장하는 주장은 다음 네 종류로 구분한다. 주장 성격과 명제의 근거 상태는 별도의 축으로 관리한다.

| 주장 성격 | 의미 | 필수 기록 |
|---|---|---|
| `FACT` | 공개 원문이나 제공된 데이터에서 직접 확인된 사실 | 출처 ID와 적용 범위 |
| `EXPERIENCE` | 사용자가 직접 겪거나 관찰한 사례 | 증명 가능한 범위와 일반화 한계 |
| `INFERENCE` | 확인된 사실·경험에서 논리적으로 도출한 해석 | 사용한 항목 ID와 추론 과정 |
| `ASSUMPTION` | 아이디어 성립에 필요하지만 아직 확인되지 않은 주장 | 확인 방법과 실패 조건 |

개인 경험 하나만으로 문제의 빈도, 전체 사용자 비율, 원인 또는 사업 효과를 일반화하지 않는다.

`UNKNOWN`은 주장 종류가 아니라 명제를 판정하기 위한 근거가 부족한 상태다.

### 출처 추적성

모든 외부 사실에는 출처 ID를 부여하고 문서 제목, 발행 주체·저자, 발행·수정일, 확인일, 원문 URL, 근거 위치와 적용 한계를 기록한다. 명제는 판단에 사용한 사실·경험·추론·가정 ID를 참조해야 한다.

검색 결과 요약만 확인했거나 원문에 접근하지 못한 자료는 조사 후보로만 남긴다. 이를 확인된 사실이나 최종 권고의 근거로 사용하지 않는다.

### 명제 판정 상태

각 명제에는 다음 중 하나의 상태를 부여한다.

| 상태 | 기준 |
|---|---|
| `SUPPORTED` | 명제를 직접 지지하는 근거가 있고 중요한 반대 근거가 없음 |
| `PARTIAL` | 일부 범위만 확인됐거나 근거가 간접적임 |
| `CONTRADICTED` | 명제를 약화하거나 반박하는 직접 근거가 있음 |
| `UNKNOWN` | 판단에 필요한 근거가 부족함 |

`UNKNOWN`은 실패가 아니라 다음 확인 작업이 필요하다는 의미다. 에이전트는 자료를 찾지 못했다는 이유만으로 `CONTRADICTED`를 부여하지 않는다.

### 명제별 기록 형식

```text
명제 ID:
명제:
명제 유형:
틀렸다고 판단할 수 있는 조건:
지지 항목 ID:
반대 항목 ID:
미검증 가정 ID:
현재 상태:
판단 이유:
다음 확인 행동:
```

### 아이디어 판단과의 연결

아이디어 결과는 명제 상태의 단순 평균으로 결정하지 않는다. 대상·문제·중요성처럼 아이디어의 존재 이유를 구성하는 핵심 명제가 `UNKNOWN`이거나 `CONTRADICTED`라면 바로 `GO`로 판단하지 않는다.

에이전트는 다음 순서로 판단한다.

1. 핵심 명제를 식별한다.
2. 각 명제의 근거 상태를 확인한다.
3. 가장 약한 핵심 명제를 찾는다.
4. 그 명제를 확인할 가장 작은 행동을 제안한다.
5. 현재 상태에서 정식 기획으로 발전시킬 가치가 있는지 권고한다.
