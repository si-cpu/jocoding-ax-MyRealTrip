# Project Charter: 여행 콘텐츠 키워드 지도

- 문서 버전: 0.6.1
- 상태: 승인
- 작성일: 2026-07-06
- 대상 기업: 마이리얼트립
- 제품 형태: Codex 플러그인
- 주요 사용자: 목적지는 정했지만 현지에서 무엇을 하고 먹을지 아직 정하지 못한 여행자
- 한 줄 정의: 여행지를 입력하면 마이리얼트립의 예약 가능한 상품을 분석하여 `뭐 하지?`와 `뭐 먹지?`를 구체적인 콘텐츠 키워드로 펼쳐주고, 선택한 키워드와 관련된 공식 상품으로 연결한다.

## 1. 프로젝트 개요

### 1.1 배경

여행 상품 플랫폼의 기존 분류는 대체로 투어, 입장권, 액티비티, 교통처럼 **상품의 형식**을 설명한다. 그러나 목적지를 정한 여행자가 처음 품는 질문은 “어떤 유형의 상품을 살까?”보다 “그곳에서 뭐 하지?”와 “뭐 먹지?”에 가깝다.

마이리얼트립 상품에는 실제로 장소명, 작품명, 스포츠팀, 공연명, 음식명과 같은 구체적인 여행 콘텐츠가 포함되어 있다. 하지만 이러한 정보는 여러 상품의 제목·설명·코스 안에 흩어져 있다. 본 프로젝트는 새로운 상품을 추천하는 대신, 기존 상품에 포함된 구체적인 경험을 여행자의 언어로 다시 색인화한다.

### 1.2 문제 정의

> 목적지를 정한 여행자는 그곳에서 할 일과 먹을 일을 구체적으로 떠올리지 못해 여러 상품을 반복해서 열어봐야 하며, 여행 목적과 예약할 상품을 정하는 데 시간이 걸린다.

### 1.3 해결 개념

`여행지 입력 → 뭐 하지?/뭐 먹지? 키워드 지도 → 키워드 선택 → 관련 상품 목록 → 마이리얼트립 공식 상품 페이지`

예를 들어 `오사카`를 입력하면 다음과 같이 보여준다.

- 뭐 하지?: 유니버설 스튜디오 재팬, 슈퍼 닌텐도 월드, 오사카성, 쿠로몬시장
- 뭐 먹지?: 오코노미야키, 타코야키, 모던야키, 쿠시카츠

`애니메이션`, `음식`, `명소`처럼 넓은 표현보다 `해리포터`, `나루토`, `오코노미야키`, `한신 타이거스`처럼 여행 목적을 바로 떠올릴 수 있는 고유명사와 구체적인 행동을 우선한다.

## 2. Project Purpose

### 2.1 목적

마이리얼트립에서 예약 가능한 상품 안에 흩어진 경험 정보를 구체적인 콘텐츠 키워드로 재구성하여, 여행자가 목적지에서 가능한 일을 빠르게 발견하고 관련 상품 탐색으로 이동하게 한다.

### 2.2 기대 가치

#### 여행자

- 긴 상품 목록을 읽기 전에 해당 여행지에서 가능한 경험의 전체 윤곽을 파악한다.
- AI가 고른 소수 추천에 갇히지 않고 스스로 관심 키워드를 선택한다.
- 선택한 경험과 연결되는 예약 상품으로 바로 이동한다.

#### 마이리얼트립

- 기존 상품 데이터를 새로운 탐색 단위로 재활용한다.
- 상품 설명 안에 숨어 있던 장소·음식·작품·공연·팀 정보를 탐색 진입점으로 만든다.
- 향후 도시별 랜딩 페이지, SNS, 뉴스레터 등 콘텐츠 제작에 키워드 지도를 재사용할 가능성을 검토할 수 있다.

콘텐츠 제작과 개인화는 파생 가치이며 MVP의 주목적은 아니다.

## 3. 사실, 관찰 및 가설

### 3.1 공개 자료로 확인된 사실

1. 마이리얼트립은 도시별로 투어·티켓 상품과 카테고리를 제공한다.
2. 개별 상품 페이지에는 상품명, 지역, 설명, 일정·코스, 가격, 예약 링크 등 경험을 구체화할 수 있는 정보가 포함된다.
3. 마이리얼트립 개발자 인터페이스를 통해 상품을 조회하고 공식 상품 URL로 연결할 수 있다.
4. 같은 상품 안에도 상품 유형보다 구체적인 다수의 콘텐츠가 존재한다. 예를 들어 하나의 부산 투어에 스카이캡슐, 감천문화마을, 흰여울문화마을이 함께 포함될 수 있다.

### 3.2 현재 제품에 대한 관찰

공개 화면에서 도시별 상품 목록, 상품 유형 카테고리, 인기 상품은 확인했다. 반면 다음의 완전한 흐름은 확인하지 못했다.

> 여행지 → 뭐 하지?/뭐 먹지? → 작품명·음식명·장소명·팀명 등 구체적인 키워드 → 관련 예약 상품 묶음

따라서 “마이리얼트립에 없는 정보”가 아니라 **기존 상품 정보를 다르게 탐색하는 방식**으로 정의한다. 공개 화면 전체에 기능이 전혀 없다고 단정하지 않는다.

### 3.3 검증할 핵심 가설

| ID | 명제 | 상태 |
|---|---|---|
| H1 | 상품 유형보다 구체적인 경험 키워드가 여행지에서 가능한 일을 더 빠르게 이해하게 한다. | 미검증 |
| H2 | `뭐 하지?`와 `뭐 먹지?`를 분리하면 사용자가 원하는 탐색 목적을 더 빠르게 선택한다. | 미검증 |
| H3 | 키워드와 관련 상품을 연결하면 일반 상품 목록보다 상품 상세 진입까지 필요한 탐색 시간이 줄어든다. | 미검증 |
| H4 | 상품에 직접 명시된 내용만 키워드로 사용하면 AI의 과도한 추론과 잘못된 상품 연결을 통제할 수 있다. | 로직 검증 필요 |
| H5 | 여행지별 콘텐츠 키워드가 축적되면 향후 동의 기반 개인화와 콘텐츠 제작의 기초 데이터가 된다. | 장기 가설 |

## 4. 목표와 성공 기준

### 4.1 MVP 목표

1. 사용자가 여행지 하나만 입력해도 결과를 받을 수 있다.
2. 결과를 `뭐 하지?`와 `뭐 먹지?`로 명확히 분리한다.
3. 결과를 여행 목적이 그려지는 구체적인 키워드로 표시한다.
4. 각 키워드에 근거가 된 마이리얼트립 상품을 연결한다.
5. 공식 상품에서 확인되지 않은 정보는 생성하지 않는다.

### 4.2 1차 성공 기준

MVP의 성공을 결제 전환으로 단정하지 않는다. 우선 다음 사용자 행동을 검증한다.

- 사용자가 이전에 몰랐던 경험 키워드를 하나 이상 발견한다.
- 결과를 본 뒤 하고 싶거나 먹고 싶은 항목을 하나 이상 선택한다.
- 키워드에서 관련 상품 상세 페이지로 이동한다.
- 기존 상품 목록 대비 첫 관심 상품을 찾는 시간이 감소한다.
- 잘못된 도시·콘텐츠·상품 연결이 사전 기준 이하로 유지된다.

해커톤 MVP에서는 일정상 개발자 자체 테스트로 기능 정확성만 확인한다. 실제 사용자 행동 개선 여부는 후속 검증으로 남기며, 제출물에서 입증된 효과로 주장하지 않는다.

## 5. Scope

### 5.1 In Scope — MVP

- 사용자가 입력한 여행지의 정규화 및 상품 검색 범위 결정
- 마이리얼트립에서 조회 가능한 투어·티켓·체험·식사 관련 상품 수집
- 상품명, 설명, 코스에서 구체적인 콘텐츠 후보 추출
- 장소명, 시설명, 작품명, 캐릭터명, 스포츠팀, 공연명, 음식명, 구체적 활동 추출
- 동의어와 표기 차이 통합
  - 예: `USJ`와 `유니버설 스튜디오 재팬`
- `뭐 하지?`와 `뭐 먹지?` 분리
- 키워드별 관련 상품 묶음
- 각 상품의 공식 마이리얼트립 링크 제공
- 근거가 부족하거나 연결할 상품이 없는 항목의 제외 또는 별도 표시
- API 실패·빈 결과·모호한 지역 입력에 대한 안전한 응답

### 5.2 Out of Scope — MVP

- 항공권·숙소 추천 및 완성 일정 생성
- 사용자에게 가장 적합한 상품의 순위 결정
- 성격, 소득, 동행 형태 등 숨은 사용자 특성 추론
- 예약 이력이나 개인정보의 자동 조회·저장
- 예약·결제·취소의 대리 실행
- 마이리얼트립에 없는 관광지·맛집의 포괄적 여행 가이드 제공
- 상품의 품질, 최저가, 재고 또는 예약 성공 보장
- 도시별 마케팅 콘텐츠의 자동 발행
- 사용자 행동 데이터 기반 개인화 모델 구축

### 5.3 향후 확장 후보

- 사용자가 명시적으로 선택한 키워드에 기반한 세션 내 필터링
- 동의한 사용자에 한해 저장·클릭·예약 키워드를 활용한 개인화
- 이미 경험한 콘텐츠와 새로운 콘텐츠의 구분
- 계절, 운영일, 예약 가능 날짜 기반 필터
- 키워드 지도를 활용한 도시별 콘텐츠 초안 생성

확장 기능은 MVP 검증 이후 별도 승인한다.

## 6. 사용자 경험 및 기능 요구사항

### 6.1 기본 입력

- 필수: 여행지
- 선택: 여행 날짜
- 선택: `뭐 하지?`, `뭐 먹지?` 또는 모두

취향 입력은 필수가 아니다. 개인 데이터가 없어도 전체 기능이 작동해야 한다.

### 6.2 기본 출력

1. 인식한 여행지
2. `뭐 하지?` 키워드
3. `뭐 먹지?` 키워드
4. 선택한 키워드와 관련된 상품 목록
5. 상품명, 확인된 연관 근거, 공식 링크
6. 예약 가능 여부를 확인할 필요가 있다는 안내

### 6.3 키워드 품질 기준

포함 우선순위:

1. 작품·캐릭터·프랜차이즈: 해리포터, 슈퍼 닌텐도 월드
2. 스포츠팀·공연: 아스널, 한신 타이거스, 위키드
3. 고유 장소·시설: 스톤헨지, 리즈바역, 감천문화마을
4. 구체적 음식: 오코노미야키, 징기스칸, 충칭 훠궈
5. 장면이 그려지는 행동: 광안대교 야경 요트, 한복 스냅 촬영

제외 우선순위:

- 너무 넓은 말: 관광, 체험, 음식, 액티비티
- 판매 문구: 인기, 추천, 특가, 필수, 프리미엄
- 상품 형식만 나타내는 말: 일일투어, 입장권, 단독투어
- 근거 없는 연상: 애니메이션 상품을 특정 작품명으로 임의 연결

### 6.4 상품 연결 규칙

- 키워드가 상품명·설명·코스에 직접 등장하거나 명확히 동일한 대상일 때만 연결한다.
- 하나의 키워드에 여러 상품이 연결될 수 있다.
- 하나의 상품에서 여러 키워드를 추출할 수 있다.
- 검색 결과에 없는 상품 URL을 생성하지 않는다.
- 상품 페이지가 존재해도 선택 날짜의 판매 여부는 별도로 확인하게 한다.
- 먹거리 상품은 식사권, 식당 예약, 푸드투어, 쿠킹클래스, 식사가 포함된 투어를 우선한다.
- 음식이 단순 안내 정보일 뿐 결제 대상이 아니면 `예약 상품 연결 없음`으로 구분하거나 MVP 결과에서 제외한다.

## 9. 운영 일정

정확한 마감일과 가용 시간에 따라 날짜는 조정하되, 작업 순서는 다음과 같이 고정한다.

| 단계 | 주요 작업 | 완료 조건 |
|---|---|---|
| 1. Charter 확정 | 문제, 사용자, 범위, 가설, 위험 합의 | 본 문서 승인 |
| 2. 데이터 탐색 | API 응답 필드와 도시별 상품 표본 확인 | 최소 5개 도시의 원본 상품 표본 확보 |
| 3. 분류 규칙 설계 | 키워드 포함·제외·동의어·근거 규칙 작성 | 사람이 동일 입력을 재현 가능 |
| 4. MVP 구현 | 조회, 추출, 그룹화, 링크 출력 구현 | 대표 도시 입력이 끝까지 동작 |
| 5. 기능 검증 | 정상·예외·오인 연결 테스트 | 치명 오류 0건, 알려진 한계 기록 |
| 6. 자체 시나리오 검증 | 대표 도시·빈 결과·오인 연결·API 실패 점검 | 사전 정의한 자체 테스트 통과 |
| 7. 제출 정리 | README, 질문지, 로그, ZIP 정합성 확인 | 제출 구조 및 실행 검증 완료 |

운영 이후 데이터 자동 갱신, 모니터링, 개인화는 해커톤 MVP 일정에 포함하지 않는다.

## 10. Governance and Decision Rules

- Charter의 핵심 문제·사용자·범위를 변경하면 버전을 올리고 변경 이유를 기록한다.
- 공개 근거로 확인된 사실과 프로젝트 가설을 문서에서 분리한다.
- 기능 추가는 H1~H4 검증에 필요한 경우에만 MVP에 포함한다.
- 결과가 그럴듯해 보여도 상품 근거가 없으면 제외한다.
- 자동화가 모호한 경우 더 많은 키워드를 보여주기보다 근거 있는 적은 키워드를 선택한다.
- 최종 예약 판단과 결제는 항상 사용자가 수행한다.

## 12. 검증 개요

### MVP 검증 범위

이번 제출은 개인 개발과 잔여 일정을 고려해 개발자 자체 테스트만 수행한다.

- 단위 테스트: 페이지 순회, 도시 필터, 카테고리 제외, 상세 정리, 날짜 옵션 판정
- 정적 검증: JSON·YAML·Skill·Plugin 구조, 비밀정보, 문법, 파일 구조
- 시나리오 검증: 대표 도시, 상품 부족 도시, 지역 입력, API 키 부재, 상세 누락

자체 테스트는 기능이 설계대로 동작하는지만 확인한다. 탐색 시간 감소, 선택 확신, 상세 클릭 또는 결제 전환 같은 사용자 효과는 입증하지 않는다.

### 후속 사용자 검증

### 비교 조건

- A: 도시의 기존 상품 목록 또는 상품 유형 중심 탐색
- B: 동일 도시의 `뭐 하지?·뭐 먹지?` 콘텐츠 키워드 탐색

### 과업

“이 도시에서 하고 싶거나 먹고 싶은 경험 하나를 고르고 예약할 상품 상세 페이지까지 이동하세요.”

### 관찰 지표

- 첫 관심 키워드 선택까지 걸린 시간
- 상품 상세 진입까지 걸린 시간
- 새롭게 발견한 경험 수
- 선택한 경험을 여행 목적으로 설명할 수 있는지
- 잘못 연결되었다고 판단한 키워드·상품 수
- 탐색 만족도와 선택 확신

### 중단 또는 수정 기준

- 잘못된 도시나 관련 없는 상품 연결이 반복된다.
- 키워드가 기존 카테고리보다 구체적이지 않다.
- 대부분의 키워드에 연결 가능한 상품이 없다.
- 사용자에게 키워드 지도가 기존 상품 목록보다 빠르거나 이해하기 쉽지 않다.

이 비교 실험은 MVP 제출 이후 수행할 후속 검증이다.

## 13. 개인정보 및 개인화 원칙

MVP는 여행지 입력만으로 동작하며 개인정보와 예약 이력을 요구하지 않는다.

향후 개인화를 도입할 경우 다음 순서를 따른다.

1. 현재 세션에서 사용자가 직접 선택한 키워드로만 결과를 좁힌다.
2. 저장 기능은 명시적인 동의를 받은 경우에만 제공한다.
3. 수집 목적, 저장 기간, 삭제 방법을 먼저 알린다.
4. 민감한 성향이나 숨은 취향을 추론하지 않는다.
5. 개인화를 끈 사용자도 기본 키워드 지도를 동일하게 사용할 수 있게 한다.

## 15. 승인 기준

다음 항목에 합의하면 Charter를 확정한다.

- 핵심 문제는 `여행지에서 뭐 하지?·뭐 먹지?를 빠르게 파악하기 어렵다`이다.
- 주요 사용자는 마이리얼트립 직원이 아니라 여행자이다.
- 해결책은 추천 순위가 아니라 구체적인 콘텐츠 키워드 지도이다.
- 공식 마이리얼트립 상품에 근거한 키워드만 예약 상품과 연결한다.
- MVP는 투어·티켓·체험·먹거리 탐색에 한정한다.
- 개인화, 콘텐츠 자동 제작, 예약 실행은 후속 범위이다.
