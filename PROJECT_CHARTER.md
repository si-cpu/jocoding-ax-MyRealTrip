# Project Charter: 새로운 나를 찾는 여행

- 문서 버전: 0.5
- 작성일: 2026-07-03
- 대상 기업: 마이리얼트립
- 플러그인 버전: 0.4.0
- 한 줄 정의: 첫 화면에서 세 사용법을 안내하고, 공항에서 어디로 갈까·어디서 잘까·오늘 또는 내일 뭐 하지 요청에 공식 마이리얼트립 데이터로 답하는 Codex 플러그인

## 1. 목적

일반적인 개인화는 사용자가 좋아하던 선택을 더 잘 찾아준다. 이 프로젝트는 반대로 사용자가 직접 밝힌 익숙함만 잠시 제외해, 평소라면 검색하지 않았을 여행 선택지를 발견하도록 넛지한다.

플러그인은 발견과 설명까지만 담당한다. 예약과 최종 판단은 사용자가 직접 한다.

> 늘 고르던 여행 밖에 무엇이 있는지 보여준다.

## 2. 확인된 사실과 핵심 가설

### 확인된 사실

1. 마이리얼트립은 항공·숙박·투어·교통·티켓을 제공하고, 고객은 이 요소를 조합해 자유여행을 만든다.
2. 마이리얼트립은 외부 개발자에게 상품 데이터용 Partner REST API와 MCP를 제공한다.
3. 항공 API는 공항 자동완성, 숙소 API는 지역 자동완성, 투어 API는 도시별 카테고리를 통해 검색 식별자를 구조화한다.
4. 각 API의 식별자와 페이지 규칙은 서로 다르므로 하나의 모호한 키워드 검색으로 합치면 오인 가능성이 있다.

### 검증할 핵심 가설

새로운 경험을 원하는 여행자에게 익숙한 목적지·숙소 유형·취미 관련 투어를 제외한 실제 후보를 보여주면, 일반적인 취향 강화 추천보다 새로운 선택지 발견이 늘어날 것이다.

이 가설은 공개 자료로 직접 입증되지 않았다. 플러그인의 검증 대상이지 사실이 아니다.

## 3. 하나의 사용법, 세 개의 명확한 입구

첫 실행에서는 다음 세 사용법과 예시를 먼저 보여준다. 사용자는 내부 명령이나 별도 스킬을 알 필요 없이 자연어로 요청한다.

- `공항에서 어디로 갈까`: 출발 공항에서 갈 수 있는 익숙하지 않은 해외 목적지
- `어디서 잘까`: 여행지에서 익숙한 숙소 유형을 제외한 숙소
- `오늘·내일 뭐 하지`: 해당 날짜에 실제 예약 가능한 낯선 투어·티켓

### 입구 A. 공항에서 어디로 갈까

- 입력: 출발 공항, 3~7일의 여행 기간, 익숙한 목적지
- 범위: 전체 목적지 최저가 API가 지원하는 국제선
- 공식 데이터: 공항 자동완성, 전체 목적지 캘린더 최저가, 선택 노선의 실시간 항공편 검색
- 출력: 익숙한 도시·국가를 제외한 최대 3개의 목적지와 캘린더 최저가
- 후속 동작: 사용자가 목적지를 고르면 반환된 날짜로 실제 항공편과 예약 링크를 다시 확인
- 주의: 캘린더 최저가는 실시간 좌석이나 결제 가격이 아니다.

### 입구 B. 어디서 잘까

- 입력: 여행지, 날짜, 익숙한 숙소 유형
- 공식 데이터: REST 지역 자동완성·숙소 검색, MCP 숙소 유형 검색
- 제외: 호텔·호스텔·료칸 등 사용자가 익숙하다고 밝힌 유형
- 출력: 지역·날짜·숙소 유형이 검증된 다른 형태의 숙소
- 주의: REST 숙소 응답에는 숙소 유형이 없으므로 MCP의 구조화된 유형과 교차 확인한다.

### 입구 C. 오늘·내일 뭐 하지

- 입력: 현재 도시, 오늘 또는 내일, 평소 취미
- 공식 데이터: 투어 카테고리·검색·상세·날짜별 옵션
- 제외: 취미와 직접 또는 명확하게 겹치는 투어
- 출력: 지역이 일치하고 취미와 겹치지 않으며 해당 날짜에 옵션이 존재하는 투어·티켓
- 주의: 검색 시작가만으로 예약 가능하다고 표현하지 않는다.

## 4. 오인 방지 규칙

### 항공

- `airport.code`만 사용한다.
- `city.code`를 공항코드로 사용하지 않는다.
- 사람이 읽는 도시·국가와 IATA 코드를 함께 표시한다.
- 전체 목적지 최저가는 국제선 캘린더 데이터로만 표현한다.
- 캘린더 가격과 선택 후 실시간 항공편 가격을 섞지 않는다.
- 목적지 코드를 공항 자동완성으로 해석하지 못하면 사용자에게 노출하지 않는다.

### 숙소

- 지역 자동완성에서 정확히 일치하는 `CITY`의 `regionId`만 사용한다.
- 역·공항·관광지 또는 비슷한 이름의 다른 지역을 자동 선택하지 않는다.
- 숙소명에 포함된 ‘호텔’ 같은 단어가 아니라 구조화된 숙소 유형으로 판정한다.

### 투어

- 검색 결과의 `description` 지역과 요청 도시를 일치시킨다.
- 제목에 다른 도시가 명시되면 제외한다.
- 지역 검증을 통과한 뒤 취미 겹침을 판정한다.
- `options`가 비어 있으면 선택 날짜에 예약 불가로 처리한다.
- 날짜별 옵션 가격을 검색 시작가보다 우선한다.

## 5. 공통 원칙

- 사용자가 직접 밝힌 익숙함만 제외한다.
- 성격·심리·숨은 취향·여행 이력을 추론하지 않는다.
- 불명확한 관계를 억지로 ‘반대 취향’으로 만들지 않는다.
- 공식 응답에 없는 상품·가격·평점·재고·링크를 생성하지 않는다.
- 후보가 부족하면 제외 범위를 몰래 완화하지 않는다.
- API 키는 `MYREALTRIP_API_KEY` 환경변수로만 읽고 대화·코드·로그에 남기지 않는다.
- 예약·결제·취소를 대신하지 않는다.

## 6. MVP 범위

### 포함

- 하나의 스킬과 사용법으로 필요한 검색 분기를 자동 선택
- 공식 API 식별자를 이용한 지역·공항 검증
- 국제선 전체 목적지 캘린더 최저가를 이용한 목적지 발견
- 익숙한 선택의 설명 가능한 제외
- 공식 인터페이스가 사용 조건을 검증한 실제 상품만 제시
- 공식 상품 링크가 제공되는 경우 그대로 연결

### 제외

- 항공·숙소·투어를 자동으로 묶은 완성 일정
- 사용자 계정 또는 예약 이력 조회
- 결제·예약·취소 자동화
- 최저가 또는 예약 가능성 보장
- 자기 발견이나 행동 변화 보장

## 7. 검증 명제

| ID | 명제 | 현재 상태 |
|---|---|---|
| H1 | 익숙한 선택을 제외하면 새로운 후보 발견이 증가한다. | 미검증 |
| H2 | 제외 이유를 설명하면 결과가 무작위가 아닌 의도된 넛지로 이해된다. | 미검증 |
| H3 | 구조화된 식별자를 사용하면 키워드 검색의 지역·공항 오인을 줄일 수 있다. | 로직 검증, 실데이터 추가 검증 필요 |
| H4 | 익숙한 목적지를 제외한 뒤에도 실제 항공 후보가 남는다. | 실제 Partner API 검증 필요 |

## 8. 검증 방법

- A: 익숙한 선택을 강화한 일반 추천
- B: 동일 조건에서 익숙한 선택을 제외한 추천

관찰 항목:

- 처음 알게 된 후보 수
- 상세 링크 또는 후속 실시간 항공 검색으로 이동한 후보 수
- 실제 고려 의향
- 지역·공항·유형 오인 건수
- `의도적으로 새롭다`와 `근거 없이 엉뚱하다` 평가

표본과 통과 수치는 테스트 전에 정한다. 결과를 본 뒤 기준을 바꾸지 않는다.

## 9. 성공의 정의

1차 성공은 예약 전환이 아니다. 사용자가 평소라면 검색하지 않았을 공식 마이리얼트립 후보를 발견하고 하나 이상을 진지하게 검토하는 것이다.
